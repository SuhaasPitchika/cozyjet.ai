# CozyJet — Complete Redesigned Build Prompt

---

## The Idea

CozyJet is a solo AI social media manager built for developers, designers, freelancers, consultants, and digital creators who are too busy doing great work to market it consistently. The core premise is simple: you connect your existing tools once during onboarding, and CozyJet silently monitors them in the background, builds a live feed of everything you have been working on, and whenever you want to create content you open the app, pick what to post about, and three platform-ready variations are generated in under thirty seconds. No screen watching, no waiting, no productivity scoring, no surveillance. CozyJet sits invisibly in the background like a marketing employee who already knows your work, your voice, and your audience — and springs into action the moment you need content. The three agents are Skippy (the background data connector that silently reads your connected tools), Snooks (the content calendar manager and strategist who tells you what to post and when), and Meta (the AI copywriter who writes the actual content in your voice across every platform simultaneously). Together they replace what a social media manager, a copywriter, and a content strategist would do for a brand — but for a solo creator, automatically, affordably, and with zero team required.

## Dashboard Frontend

The dashboard must be built in Next.js 14 with the App Router, TypeScript, Tailwind CSS, Framer Motion for animations, Radix UI for accessible primitives, TanStack Query v5 for server state, and Zustand for client state. The visual style must be clean, elegant, and minimal — a dark sidebar on the left, a clean white or near-white main content area, pixel-perfect spacing, and a consistent design system using a single accent color (deep indigo or electric blue works well) against neutral grays and whites. Every font must feel premium and readable. No clutter, no overwhelming dashboards, no feature overload visible at once.

The persistent left sidebar contains the CozyJet logo at the top, five navigation links (Create, Calendar, Analytics, Tune, Settings), a live agent status section showing Skippy, Snooks, and Meta as colored dots (green when active, gray when idle), a recent activity feed showing the last three things that happened (new content seed detected, post scheduled, post published), and the user avatar with name and plan tier at the bottom. The sidebar is collapsible on smaller screens.

The **Create page** is the heart of the product and must load with two panels side by side. The left panel is the Skippy Feed — a reverse-chronological list of content seeds automatically detected from the user's connected tools. Each seed card shows the source icon (GitHub, Notion, Figma, Google Drive, Google Calendar), the title of the work detected, a one-line AI-generated summary of what it is, the time detected, and two buttons: Generate Content and Dismiss. Above the feed is a manual input bar where users can type or speak anything they want to turn into content, with a microphone icon that activates browser speech-to-text. The right panel is the Meta generation workspace. When a seed is selected or manual input is submitted, this panel shows platform toggle buttons (LinkedIn, Twitter, Instagram, YouTube, Reddit), a Generate button, and after generation renders three variation cards per platform in a tabbed interface. Each variation card shows the full content text, a character count, a copy button, a Refine button that opens an inline chat to ask Meta to adjust the tone or angle, a Schedule button, and a Publish Now button.

The **Calendar page** renders a monthly grid where each cell shows colored dots per scheduled post. Clicking any day opens a side panel listing all content for that day with platform icons, preview text, and edit or delete options. Below the calendar a Snooks Suggestions strip shows five AI-recommended content ideas for the current week based on what Skippy has collected and what is trending, each with a one-click Generate button.

The **Analytics page** shows four top-level KPI cards (posts this month, average engagement rate, total reach, top performing platform) followed by a Recharts line graph of engagement over thirty days, a bar chart comparing performance by content type, and a table of the ten best performing posts with a Repurpose button on each.

The **Tune page** is the voice and style training area described in detail in the Tuning section below.

The **Settings page** handles connected integrations, notification preferences, posting time preferences, and subscription management.

## Backend Architecture and Stack

The backend is a FastAPI Python application with PostgreSQL as the primary database, Redis for caching and background job queuing, and Celery for all scheduled and background tasks. The full dependency list to install via `pip install -r requirements.txt` is: `fastapi`, `uvicorn[standard]`, `gunicorn`, `sqlalchemy[asyncio]`, `asyncpg`, `alembic`, `redis`, `celery[redis]`, `langchain`, `langchain-anthropic`, `langchain-google-genai`, `langchain-openai`, `anthropic`, `google-generativeai`, `openai`, `pydantic`, `pydantic-settings`, `python-jose[cryptography]`, `passlib[bcrypt]`, `python-multipart`, `httpx`, `websockets`, `pillow`, and `python-dotenv`. Required environment variables stored in `.env` and never committed to GitHub are: `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`, `OPENAI_API_KEY`, `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET_KEY` (generate with `openssl rand -base64 32`), `FRONTEND_URL`, `ALLOWED_ORIGINS`, and `ENVIRONMENT`.

The project structure follows strict separation of concerns: `app/main.py` is the FastAPI entry point, `app/config.py` holds Pydantic settings, `app/database.py` manages async PostgreSQL sessions, `app/models/` contains all SQLAlchemy ORM models, `app/schemas/` holds all Pydantic request and response shapes, `app/api/` contains route handlers organized as `auth.py`, `skippy.py`, `snooks.py`, `meta.py`, `analytics.py`, and `tune.py`, `app/agents/` contains LangChain agent definitions for each of the three agents, `app/tasks/` contains all Celery background task definitions, `app/services/` contains business logic decoupled from route handlers, and `app/dependencies.py` holds the JWT authentication dependency injected into every protected route.

The database schema requires the following tables. The `users` table stores id as UUID primary key, email as unique indexed string, password_hash as bcrypt string, display_name, avatar_url, subscription_tier as enum (free, pro, studio), voice_profile as JSON containing tone, style, emoji_usage, formality, humor, preferred_platforms, writing_examples as a JSON array, and created_at and updated_at timestamps. The `integrations` table stores id, user_id as foreign key, platform as enum (github, notion, figma, google_drive, google_calendar, linkedin, twitter, instagram), access_token encrypted at rest, refresh_token encrypted at rest, token_expires_at, is_active as boolean, last_synced_at, and created_at. The `content_seeds` table stores id, user_id, title, description as text, source_platform, source_url as nullable string, source_metadata as JSON, tags as text array, is_used as boolean, and created_at. The `content` table stores id, user_id, seed_id as nullable foreign key, platform, content_type, content_text, variation_index as integer 0 through 2, status as enum (draft, approved, scheduled, published, failed), scheduled_time, published_at, platform_post_id, error_message, and created_at plus updated_at. The `post_analytics` table stores id, content_id, platform, views, likes, comments, shares, engagement_rate as float, and fetched_at. The `tune_samples` table stores id, user_id, sample_text as text, platform, source as enum (manual, imported, approved_post), and created_at.

All protected routes must use a `get_current_user` FastAPI dependency that extracts the Bearer token, decodes it with `python-jose`, queries the user from PostgreSQL, and returns the user object or raises HTTP 401. Rate limiting must use Redis sliding window: five requests per minute on auth endpoints, ten per minute on AI generation endpoints, and one hundred per minute on all others. CORS must whitelist only the frontend origin.

## The Three Agents

**Skippy — Background Data Connector**

Skippy runs as a Celery background task, not as a screen watcher or active process the user must start. When a user connects an integration, Skippy's sync task is registered in Celery Beat to run on a schedule — every two hours for GitHub, Notion, Figma, and Google Drive, and every six hours for Google Calendar. The Celery task `skippy_sync_user_integrations` queries all active integrations for a given user, calls each platform's API using the stored OAuth token via httpx, fetches recent activity (last ten GitHub commits, last five Notion pages updated, last three Figma files modified, files added or updated in Google Drive in the last twenty-four hours, calendar events completed today), and for each item of detected activity calls the Skippy LangChain agent with Claude Sonnet to generate a one-paragraph content seed description explaining what work was done, why it matters, and what audience it would resonate with, then saves a new `content_seeds` record and sends a WebSocket notification to the user's dashboard so the feed updates in real time without a page refresh.

Skippy also provides an on-demand analysis endpoint at `POST /api/skippy/analyze-now` that triggers an immediate sync outside the scheduled window, useful when a user just finished something and wants to create content immediately. A screenshot upload endpoint at `POST /api/skippy/screenshot` accepts a multipart image, sends it to GPT-4 Vision or Claude with a prompt to identify visible work and summarize it, and creates a seed. A voice input endpoint at `POST /api/skippy/voice` accepts a transcribed string from the browser's Web Speech API and passes it to the Skippy LangChain agent for enhancement and seed creation.

**Snooks — Content Calendar and Strategy Agent**

Snooks is implemented with LangChain using Google Gemini Pro as its model because Gemini excels at structured planning output. Snooks has three responsibilities. First, it generates weekly content suggestions by combining the user's recent seeds from the `content_seeds` table with trending topics fetched from the `trends` table (populated hourly by a Celery task) and the user's historical best-performing content types from `post_analytics`, then calling the Gemini agent to produce five specific content recommendations for the current week with a title, suggested platform, best posting day, and reasoning for each. This runs automatically every Sunday night via Celery Beat and also on demand at `POST /api/snooks/suggest`. Second, it calculates optimal posting times per platform by analyzing the `post_analytics` table for the user's historical engagement grouped by hour and day of week and returning the top three time slots per platform at `GET /api/snooks/optimal-times`. Third, it manages the content calendar by serving `GET /api/snooks/calendar` which returns all scheduled content for the next thirty days joined with seed information and engagement predictions.

**Meta — AI Copywriter and Content Factory**

Meta is the most user-facing agent and is implemented with LangChain using Claude Sonnet as its primary model with `ConversationBufferMemory` keyed per user ID so refinement conversations are stateful and context-aware. The primary generation endpoint at `POST /api/meta/generate` accepts a `seed_id` and a `platforms` array, loads the seed description, loads the user's full `voice_profile` from the `users` table, and constructs a LangChain prompt that instructs Meta to generate three distinct variations for each requested platform. Variation one uses emotional storytelling with a hook, a narrative arc, and a question at the end. Variation two is direct and technical, written for peers in the same field. Variation three leads with measurable outcomes and results. Each platform has hard formatting constraints built into the prompt: LinkedIn posts are 1,200 to 1,800 characters with three to five hashtags and a call to action; Twitter threads are six to ten tweets each under 280 characters with the first tweet as a strong hook; Instagram captions are conversational with line breaks and five to ten hashtags; YouTube scripts open with an eight second hook and close with a subscribe prompt; Reddit posts are detailed and community-first avoiding overt self-promotion while still showcasing the work.

Additional Meta endpoints include `POST /api/meta/generate-from-idea` for thought leadership content without a seed, `POST /api/meta/generate-from-trend` for viral trend-based content accepting a trend ID from the `trends` table, `POST /api/meta/repurpose` for transforming long-form content like blog posts or video transcripts into social posts, and `POST /api/meta/refine` which accepts a `content_id` and a natural language instruction like "make this sound more casual" or "add a stronger hook" and uses the conversation memory to revise the content while learning the user's preference.

Every time the refine endpoint is called, the service layer compares the original and revised text to detect preference signals and updates the user's `voice_profile` JSON automatically — for example if the user repeatedly asks to remove formal language the `formality` field decreases, if they keep asking to add emojis the `emoji_usage` field increases, and these learned preferences feed into every future generation making outputs progressively more accurate to the user's authentic voice.

## How Users Connect Their Apps

Integration connection happens during onboarding and is always accessible from the Settings page under Connected Apps. Each integration uses OAuth 2.0. When a user clicks Connect GitHub, the frontend redirects to GitHub's OAuth authorization URL with the required scopes (repo, read:user). After the user authorizes, GitHub redirects back to `GET /api/integrations/callback/github` with an authorization code. The backend exchanges the code for an access token and refresh token via httpx, encrypts both tokens using AES-256 before storing them in the `integrations` table, sets `is_active` to true, records `last_synced_at` as now, and immediately triggers a first Skippy sync for that integration so the user sees seeds within seconds of connecting. The same OAuth flow applies to Notion, Figma, Google Drive, Google Calendar, LinkedIn, and Twitter. Instagram requires the Facebook Graph API OAuth flow. The frontend integration settings page shows each platform as a card with a connect or disconnect button, a green or gray status dot, and the timestamp of the last successful sync. Disconnecting immediately sets `is_active` to false, cancels the Celery scheduled sync, and deletes the stored tokens. No raw OAuth tokens are ever returned to the frontend — only their connection status.

## Chat Interface

The chat interface is accessible in three places. First, an inline refinement chat appears within each generated content card on the Create page — clicking Refine opens a small chat panel below the content where the user types instructions and Meta responds with an updated version in real time, streamed token by token via the Skippy WebSocket. Second, a global floating chat button in the bottom right corner of every dashboard page opens a full side panel chat with Meta where users can ask anything — "write me a LinkedIn post about my experience with burnout," "give me five content ideas for this week," "what should I post about today based on my recent work" — and Meta responds conversationally using the user's voice profile and Snooks' suggestions as context. Third, the on-screen support chat accessible via a small floating icon is connected to a support-mode version of the Meta agent instructed to answer questions about using CozyJet, explain what the agents are doing, and help users navigate the product.

All three chat interfaces use the same WebSocket endpoint at `WS /ws/chat?token={jwt}&mode={refine|meta|support}` where the mode parameter switches the system prompt context. Messages stream back token by token so the interface feels responsive and live. Conversation history is stored per user per session in Redis with a twelve-hour TTL and optionally persisted to a `chat_sessions` table for context continuity across sessions.

## The Tune Area

The Tune page is where users teach CozyJet their exact writing voice so that every piece of generated content sounds genuinely like them rather than generic AI. The page is divided into three sections. The first section is Writing Samples where users paste three to ten examples of their best existing posts, tweets, or articles into a text area and click Analyze. The backend endpoint at `POST /api/tune/analyze-samples` saves each sample to the `tune_samples` table then calls the Meta LangChain agent with all samples concatenated and a detailed prompt asking it to identify recurring patterns in tone, vocabulary choices, sentence length, use of humor, technical depth, emoji habits, and structural preferences, and return a structured voice profile JSON. This JSON is saved to the user's `voice_profile` field in the `users` table and immediately affects all future content generation. The second section is Voice Profile where the analyzed profile is displayed as a set of labeled sliders and tags — a formality dial from casual to professional, an emoji frequency from none to frequent, a technical depth from beginner-friendly to expert-level, tone tags like honest, motivational, direct, and humorous that the user can toggle on or off, and preferred platforms showing which channels the user wants to prioritize. All changes save instantly via `PATCH /api/tune/voice-profile` and take effect immediately. The third section is Continuous Learning which explains to the user that every time they edit a generated post, approve content, or use the Refine chat, Meta silently updates their voice profile in the background. A log shows the last five automatic voice profile updates with a description like "Detected preference for shorter sentences on Twitter" or "Learned to avoid exclamation marks on LinkedIn" so the user can see the system is learning and trust it.

## Backend Connection to Frontend

The Next.js frontend stores the backend URL in `NEXT_PUBLIC_API_URL` and WebSocket URL in `NEXT_PUBLIC_WS_URL`. An Axios instance in `lib/api.ts` attaches the JWT Bearer token automatically via a request interceptor reading from Zustand auth store, and a response interceptor catches 401 errors to trigger silent token refresh at `POST /api/auth/refresh` before retrying the original request. TanStack Query manages all server state with hooks like `useContentSeeds()`, `useGenerateContent()`, `useCalendar()`, `useTrends()`, and `useAnalytics()`. The WebSocket connection is initialized in a React context provider wrapping the entire dashboard layout and pushes real-time events directly into the TanStack Query cache so the UI updates instantly when a new seed is detected, a post publishes, or Snooks adds a calendar suggestion. The Celery workers run as separate services in Docker Compose alongside the FastAPI backend, sharing the same codebase and environment variables but with different start commands. Railway deployment runs three services from the same GitHub repo: the FastAPI backend, the Celery worker, and the Celery Beat scheduler. The frontend deploys separately to Vercel with the Railway backend URL set in environment variables.
