# CozyJet — Complete Revolutionary Backend Prompt in Essay Format

---

## What You Are Building and Why It Matters

CozyJet is not a content generator and it is not a chatbot with integrations bolted on. It is an autonomous growth operator — a backend system that runs 24 hours a day making hundreds of intelligent micro-decisions about a user's social media presence while they focus entirely on their actual work. The fundamental difference between CozyJet and every other tool in this space is that CozyJet does not wait for the user to open the app and ask for something. It is always running, always watching, always preparing the next move. The backend must be built with this philosophy embedded into every architectural decision — agents are not request-response functions, they are continuously running processes with memory, goals, and autonomous decision-making capability. The system must work completely for a user with zero integrations connected because the agents can operate on the user's growth profile, niche intelligence, and conversation data alone. Integrations enhance the system but they are never a dependency for delivering value.

The stack is FastAPI with async PostgreSQL via SQLAlchemy and asyncpg, Redis for caching and Celery job queuing, Celery with Celery Beat for all scheduled autonomous tasks, OpenRouter as the primary AI gateway routing to Claude for content and writing tasks, Gemini directly for long-context strategic analysis, a lightweight fast model like Mistral 7B for classification and summarization tasks, and ElevenLabs for voice synthesis. Install everything via pip: `fastapi`, `uvicorn[standard]`, `gunicorn`, `sqlalchemy[asyncio]`, `asyncpg`, `alembic`, `redis`, `celery[redis]`, `openai` configured with the OpenRouter base URL, `google-generativeai`, `elevenlabs`, `pydantic`, `pydantic-settings`, `python-jose[cryptography]`, `passlib[bcrypt]`, `python-multipart`, `httpx`, `websockets`, `pillow`, `python-dotenv`, and `cryptography`. The environment variables required are `OPENROUTER_API_KEY`, `GEMINI_API_KEY`, `ELEVENLABS_API_KEY`, `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET_KEY`, `ENCRYPTION_KEY`, `FRONTEND_URL`, `ALLOWED_ORIGINS`, and `ENVIRONMENT`. These live in Doppler and are never hardcoded anywhere in the codebase.

## The Onboarding QnA System

The first thing a new user encounters after signup is a conversational popup that feels like talking to a smart friend who genuinely wants to understand them. This is not a form. It is not a series of dropdowns. It is a real multi-turn conversation powered by OpenRouter using Claude, handled by the `POST /api/onboarding/chat` endpoint which stores each exchange in an `onboarding_sessions` table as a JSON array of role and content message objects. The conversation continues for eight to ten exchanges and after it naturally concludes a second extraction call at temperature 0.1 parses the entire conversation history and produces a structured `growth_profile` JSON object saved to the user's record in the `users` table. This profile becomes the foundation for every single thing every agent does for this user from that moment forward.

The system prompt for the onboarding conversation must be short and must feel human. It reads: you are the CozyJet onboarding guide and your job is to understand this person deeply enough that every piece of content CozyJet creates feels like it came from their best self on their best day. Ask one question at a time. Listen to the answer before moving on. React like a real person — if they say something interesting acknowledge it before asking the next thing. Be warm and direct. Never list questions. Never use bullet points. Talk like a smart friend who genuinely wants to help them grow. Through natural conversation you need to discover their niche and sub-niche with real specificity, their current social media situation and where they feel stuck, their actual goal whether that is clients or following or credibility or product sales, their biggest fear about posting publicly, what kind of creator they admire and why, how much time per day they can realistically give, their content comfort zone, and their natural tone. After eight to ten exchanges end the conversation naturally and tell them CozyJet is ready to start working for them.

The extraction call after the conversation runs a separate API call with a prompt that takes the full conversation history as input and returns a structured JSON object with keys for `niche`, `sub_niche`, `goal`, `goal_timeline`, `current_state`, `fear`, `tone`, `content_style` as a list, `admired_creators` as a list, `time_available_minutes` as an integer, `comfort_zone`, `positioning_opportunity`, and `psychological_profile` as a paragraph summary of how this person thinks and what motivates them. The psychological profile key is the most important one and the extraction prompt must explicitly ask for it because it is what allows Meta to write content that feels genuinely human rather than generically optimized.

## Database Schema

The `users` table stores the foundation of every personalized operation in the system. Its columns are `id` as UUID primary key with server default `gen_random_uuid()`, `email` as unique indexed string, `password_hash` as bcrypt string at cost factor 12, `display_name` as nullable string, `subscription_tier` as enum with values free, pro, ultra defaulting to free, `growth_profile` as JSON containing the full extracted onboarding output, `voice_profile` as JSON containing tone, style, emoji usage, formality, humor, technical depth, and a learned preferences log that updates automatically with every content edit the user makes, `onboarding_complete` as boolean defaulting to false, `timezone` as string defaulting to UTC, and `created_at` and `updated_at` as managed timestamps.

The `integrations` table handles OAuth connections with columns `id` as UUID, `user_id` as foreign key with cascade delete, `platform` as enum covering github, notion, figma, google_drive, google_calendar, linkedin, twitter, instagram, `access_token` as text storing the AES-256 encrypted OAuth token that must never be stored in plaintext under any circumstances, `refresh_token` as encrypted nullable text, `token_expires_at` as nullable datetime, `is_active` as boolean, `last_synced_at` as nullable datetime, and `sync_error` as nullable text for the last failure message.

The `content_seeds` table stores detected and generated work opportunities with columns `id` as UUID, `user_id` as foreign key, `title` as string, `description` as text, `hook` as text storing the extracted opening line, `source_platform` as string, `source_metadata` as JSON for raw platform data, `tags` as PostgreSQL text array, `virality_score` as integer 0 through 100, `virality_reasoning` as text, `is_used` and `is_dismissed` as booleans, and `created_at` as datetime. A composite index on `(user_id, created_at DESC)` is required for fast feed loading.

The `content` table manages generated posts through their full lifecycle with columns `id` as UUID, `user_id` as foreign key, `seed_id` as nullable foreign key set null on delete, `platform` as enum, `content_type` as enum, `content_text` as text, `hook` as text, `variation_index` as integer, `status` as enum with values draft, approved, scheduled, published, failed, `virality_score` as integer, `scheduled_time` as nullable datetime, `published_at` as nullable datetime, `platform_post_id` as nullable string, `narrative_arc_id` as nullable foreign key, `arc_stage` as nullable string indicating which chapter of the story this post represents, and `error_message` as nullable text. A composite index on `(user_id, status)` and a partial index on `scheduled_time WHERE status = 'approved'` are both required.

The `conversation_opportunities` table stores Skippy's discovered conversation chances with columns `id` as UUID, `user_id` as foreign key, `platform` as string, `thread_url` as string, `thread_title` as text, `original_post` as text, `drafted_reply` as text, `relevance_score` as integer, `opportunity_reason` as text explaining why this specific user should enter this specific conversation, and `status` as enum with values pending, approved, posted, skipped.

The `relationship_targets` table powers Snooks' relationship graph with columns `id` as UUID, `user_id` as foreign key, `target_username` as string, `target_platform` as string, `target_follower_count` as integer, `alignment_score` as integer reflecting how well their audience overlaps with the user's niche, `relationship_stage` as enum with values identified, warming, engaged, connected, `engagement_count` as integer, `last_engaged_at` as nullable datetime, and `notes` as text.

The `narrative_arcs` table manages multi-week content storylines with columns `id` as UUID, `user_id` as foreign key, `arc_title` as string, `arc_description` as text, `project_context` as text, `total_stages` as integer, `current_stage` as integer, `status` as enum with values active, completed, paused, `started_at` as datetime, and `estimated_end` as datetime. The `experiments` table tracks Snooks' weekly growth experiments with columns `id` as UUID, `user_id` as foreign key, `hypothesis` as text, `content_variable` as string, `control_content_id` and `test_content_id` as UUID foreign keys, `started_at` and `ended_at` as datetimes, `result` and `learning` as nullable texts, and `applied_to_profile` as boolean tracking whether the learning has been incorporated into the voice profile.

## Model Router and API Configuration

The model router in `app/services/model_router.py` initializes three clients at module level. The OpenRouter client uses the standard `openai.AsyncOpenAI` class with `base_url` set to `https://openrouter.ai/api/v1`, the `OPENROUTER_API_KEY` as the API key, and default headers including `HTTP-Referer` set to the frontend URL and `X-Title` set to CozyJet for proper attribution. The Gemini client is initialized directly via `google.generativeai.configure` using the `GEMINI_API_KEY`. The ElevenLabs client is initialized using the official `elevenlabs.client.ElevenLabs` class with the `ELEVENLABS_API_KEY`. The router exposes four async functions: `call_openrouter(messages, model, max_tokens, temperature)` for all Meta and Skippy calls, `call_gemini(prompt, context_data)` for all Snooks strategic analysis, `call_classifier(text)` using `google/gemma-2-9b-it` at temperature 0.1 and max 100 tokens for all quick classification tasks like detecting calendar intent in chat messages, and `generate_audio(text, voice_id)` calling ElevenLabs with the `eleven_turbo_v2` model. Default model assignments are `anthropic/claude-3.5-sonnet` for Meta content generation, `mistralai/mistral-7b-instruct` for Skippy summarization and virality scoring, and `google/gemini-1.5-pro` directly for Snooks strategic analysis. Temperature assignments are 0.92 for hook generation, 0.78 for content body generation, 0.85 for onboarding conversation, 0.1 for profile extraction and classification, 0.35 for Skippy summarization, and 0.5 for Snooks strategy.

## Skippy Agent — Intelligence Layer

Skippy is defined in `app/agents/skippy_agent.py` and handles three completely separate responsibilities running simultaneously. Its system prompt is short and must be exactly this: you are Skippy, CozyJet's intelligence agent. You find what is worth sharing and where the user should show up. You think like a journalist — specific, curious, no fluff. You write seed descriptions that make someone immediately see the story worth telling. Never generic. Always specific. The prompt for seed generation from integration activity takes a JSON object containing the raw activity data from the platform, a creator context object with niche, goal, tone, and sub-niche pulled from the growth profile, and an empty output template with keys for title under fifty characters, hook as one sentence that would make someone stop scrolling, story as two sentences explaining the work and why it matters to the specific audience, why this matters to their audience as a separate field, tags as an empty array, and best platforms as an empty array. The model fills the template and returns nothing else. This strict template approach eliminates padding and generic output.

For users with no integrations connected Skippy takes whatever the user describes in natural language — typed or spoken via browser speech recognition submitted to `POST /api/skippy/voice` — and passes it through the same agent with a note in the context that this is user-described rather than integration-detected. The output quality is nearly identical because the intelligence is in the prompt and the model, not in where the raw data came from. The conversation hunting function takes the user's growth profile and their last three content items as context and asks the model to find five specific conversations happening right now where this creator has something real to add. For each opportunity it generates the platform, the topic, the specific angle this creator should take, a complete drafted reply ready to post with one tap, a reason explaining why entering this conversation would benefit this specific creator, and an estimated probability that someone reading the reply would visit the creator's profile. These opportunities are saved to the `conversation_opportunities` table and pushed to the user's dashboard via WebSocket.

Skippy also detects calendar intent in user chat messages using the classifier model. Any message containing language implying a future commitment — "I am planning to launch Friday", "I want to start posting about this next week", "remind me to share the results" — is classified at temperature 0.1 and if intent is detected the system extracts the implied date, content topic, and platform and automatically creates a `calendar_entries` record with status draft. A WebSocket notification informs the user: "Added to your calendar — topic on date for platform. Meta will draft this the day before." This happens without the user explicitly requesting it and is one of the clearest demonstrations that the system is genuinely agentic rather than reactive.

## Snooks Agent — Strategy Layer

Snooks is defined in `app/agents/snooks_agent.py` and uses Gemini 1.5 Pro directly for all calls because the strategic analysis task involves loading the user's last ninety days of post performance data, their last thirty content seeds with engagement outcomes, week-over-week growth metrics, the current top twenty trending topics in their detected niche, their full voice profile, and their stated growth goals all in a single context window. Gemini's context capacity handles this data volume natively without chunking, which is critical because strategic insights require seeing the full picture simultaneously rather than analyzing pieces in isolation. Snooks' system prompt establishes its identity as a senior personal brand strategist who thinks in systems not tactics, studies what actually works for this specific person with this specific audience, and builds repeatable strategies from real patterns. It is direct, never generic, and every recommendation must reference specific data from the user's actual history. If a suggestion would apply equally well to any random creator it is not good enough and must be regenerated.

The weekly strategy generation function produces a structured JSON output containing a seven-day content plan with each day specifying a content opportunity with title, angle, platform, optimal time based on historical engagement data, predicted engagement tier as low, medium, or high, a one-sentence reasoning that references specific past performance numbers, and urgency classification as evergreen, trending, or time-sensitive. It also produces a positioning move describing one specific action to take this week to strengthen the user's niche authority, a list of relationship actions targeting specific people in the relationship graph, a weekly experiment with hypothesis and measurement criteria, a gap opportunity identifying one content angle nobody in the user's niche is currently covering, and a strategic reasoning paragraph tying everything together. The narrative arc function takes a project description and a duration in weeks and produces a complete multi-week story structure with emotional beats — curiosity through early teaser posts, struggle through behind-the-scenes development posts, breakthrough at the key milestone, proof through results posts, and lesson through reflection posts — each stage with specific content guidance and the optimal timing for release relative to actual project milestones.

The relationship sequence function takes a specific relationship target from the `relationship_targets` table and produces a three-week engagement plan that is methodical and genuine rather than transactional. Week one involves engaging meaningfully with their content twice before any direct interaction. Week two involves one thoughtful reply to something they posted that demonstrates real familiarity with their work. Week three involves a natural direct interaction only if the previous engagement felt genuine and reciprocal. The function also identifies the most natural collaboration angle based on the alignment between the target's audience and the user's niche, and explicitly lists behaviors to avoid that would make the sequence feel calculated or awkward.

## Meta Agent — Production Layer

Meta is defined in `app/agents/meta_agent.py` and runs two completely separate API calls for every piece of content rather than generating hook and body in a single pass. The hook generation call runs at temperature 0.92 — deliberately high for creative variation — and generates six hook options for a given seed and platform. Each hook option includes the full text, the psychological mechanism it uses (curiosity gap, social proof contrast, counterintuitive claim, specific relatable pain, or bold opinion), a one-sentence explanation of why it works for this specific audience, and a platform fit score from 0 to 100. The hooks are sorted by platform fit score and the highest scoring one is passed into the body generation call as the fixed opening line. Separating these two calls with different temperatures is what produces content that opens with genuine stopping power and follows with coherent substantive body text — getting both from a single call at a compromise temperature produces mediocrity in both.

Meta's system prompt establishes its core identity and the words it must never use. It never uses dive into, game-changer, in today's world, it's important to note, delve, leverage, landscape, journey, or unlock. It writes the way a smart self-aware person writes when they are being genuinely honest and interesting. It understands that virality is not luck but specific psychology executed well and it always knows which psychological mechanism it is deploying and why. The body generation call receives the winning hook as a fixed opening line, the work story from the seed, the platform with its specific rules (LinkedIn at 1200 to 1800 characters with three to five hashtags ending in a question, Twitter as six to ten tweets each under 280 characters with the first tweet standalone, Reddit as detailed and community-first with no promotional language, Instagram at 150 to 300 characters with hashtags separate), the user's style observations from their tuning files as a list of eight specific behavioral rules like "opens paragraphs with a specific scene before making the point" rather than vague instructions like "write naturally", the user's tone from their growth profile, and if this content is part of a narrative arc the current arc stage so Meta knows whether this is a teaser post or a breakthrough post or a results post.

After generation a virality scoring call runs using Mistral 7B at temperature 0.3 which scores the complete content from 0 to 100 against the probability of performing above average in this specific niche, explains what specifically works, what specifically does not work, and provides a one-line concrete fix for the weakest element. This score and reasoning are returned alongside the content to the frontend so the user sees both the generated post and an honest pre-assessment before deciding whether to approve or improve it. The `POST /api/meta/improve-score` endpoint takes a low-scoring piece of content, passes it back to Meta with the virality reasoning as additional context, and regenerates specifically targeting the identified weakness. All platform generations run in parallel using `asyncio.gather` so total latency equals the slowest single call rather than the sum of all calls.

## Momentum Detection and the Real Time Engine

The momentum detector in `app/services/momentum_detector.py` runs as a Celery task every fifteen minutes checking all posts published within the last ninety minutes for momentum signals. It compares three metrics against the user's personal baseline calculated from their historical averages: save rate comparing current saves per impression against the user's average save rate, comment rate comparing current comments per impression against their average comment rate, and profile visit conversion comparing absolute profile visits against what this user typically sees in the first hour. When the save rate exceeds twice the baseline the momentum score increases by forty points. When the comment rate exceeds three times the baseline it increases by thirty-five points. When profile visits exceed fifty within the first sixty minutes it increases by twenty-five points. A total score above fifty within the first ninety minutes of publishing triggers a momentum alert pushed immediately via WebSocket to the user's dashboard.

The alert includes the momentum score, the specific signals detected in plain language like "save rate is two times your average — the algorithm is going to push this further", the estimated window remaining before the algorithm boost window closes, and a list of three specific amplification actions generated by the classifier model including actions like reply to every comment within the next twenty minutes to keep engagement velocity high, identify the two most influential people who engaged and follow them now while they remember you, and create a follow-up post capitalizing on the topic while the algorithm is still actively distributing the original. The user sees this alert on their phone and can approve the amplification sequence with one tap. Most creators miss their own best performing posts because they posted and moved on. This system ensures every momentum signal gets captured and amplified while the window is still open.

## Autonomous Celery Task System

The Celery application in `app/tasks/celery_app.py` uses Redis as both the broker and result backend and defines a beat schedule covering every autonomous operation in the system. The daily briefing task runs at 7 AM in each user's configured timezone and does the following without any user interaction: it triggers conversation hunting for all active users producing a fresh queue of reply opportunities, it checks whether any integrations are connected and if so triggers their sync, and it pushes a morning briefing via WebSocket using ElevenLabs to synthesize the message as audio so the user literally hears their agent speaking to them. The momentum check task runs every fifteen minutes against all posts under ninety minutes old. The publish scheduled content task runs every sixty seconds querying approved content with past scheduled times, decrypting the user's platform OAuth tokens, calling the platform publishing APIs, and updating content records with status and platform post IDs. The relationship engine task runs every morning at 9 AM identifying new relationship targets based on who engaged with recent posts, updating the stage of existing targets, and generating the next action in each active relationship sequence. The weekly strategy task runs every Sunday evening producing a fresh seven-day plan for every active user. The weekly experiment task also runs Sunday evening using Snooks to analyze the last two weeks of performance, form a hypothesis about why the top performer worked, design a test for the coming week, and save it to the experiments table. The analytics collection task runs every six hours fetching current engagement metrics from all connected platform APIs and upserting into the `post_analytics` table to keep performance data current for Snooks' analysis.

## API Endpoints and WebSocket

The authentication routes handle signup accepting email, password, and display name where the password is hashed with bcrypt at cost factor 12 and a verification token is generated and emailed, login returning a JWT access token expiring in 1440 minutes and a refresh token in an HttpOnly Secure cookie, token refresh reading from the cookie, logout blacklisting the token in Redis, and email verification. The `get_current_user` dependency in `app/dependencies.py` extracts the Bearer token, checks Redis blacklist, decodes with python-jose, queries the user, and returns the object or raises HTTP 401 — this is injected into every single protected route.

The Skippy routes include `POST /api/skippy/voice` for manual text or voice seed creation, `POST /api/skippy/screenshot` for image-based seed creation using GPT-4 Vision, `POST /api/skippy/sync-now` for immediate integration sync, and `GET /api/skippy/opportunities` returning the current conversation queue. The Snooks routes include `GET /api/snooks/strategy` for the current weekly plan, `GET /api/snooks/narrative-arcs` for active story arcs, `POST /api/snooks/create-arc` for starting a new project arc, `GET /api/snooks/relationships` for the relationship pipeline, and `GET /api/snooks/gap-report` for the competitor gap analysis. The Meta routes include `POST /api/meta/generate` running parallel platform generation with virality scoring, `POST /api/meta/refine` with conversation memory, `POST /api/meta/approve` for scheduling, and `POST /api/meta/improve-score` for automatic quality enhancement. The analytics routes include `GET /api/analytics/summary` for KPIs, `GET /api/analytics/experiments` for experiment results, and `GET /api/analytics/positioning` for brand positioning drift score.

The WebSocket endpoint at `WS /ws/main?token={jwt}` authenticates via the token query parameter, registers the connection in a Redis hash keyed by user ID, enters a receive loop handling ping messages with pong responses, and runs a background asyncio task subscribing to the Redis pub/sub channel `ws_user:{user_id}` forwarding all messages from Celery tasks to the connected client in real time. Messages pushed through this channel include new seed detected, integration connected, post published with the platform URL, momentum alert with score and actions, daily briefing with opportunity count and audio URL, relationship action required, new experiment started, and trend alert. Heartbeats run every thirty seconds and reconnection is handled with exponential backoff from two seconds doubling to sixty seconds maximum.
