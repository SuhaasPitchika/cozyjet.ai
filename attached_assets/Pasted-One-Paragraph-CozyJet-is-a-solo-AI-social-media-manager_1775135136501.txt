One Paragraph
CozyJet is a solo AI social media manager. Users connect their tools once — GitHub, Notion, Figma, Google Drive, Google Calendar — and the platform silently monitors them in the background, builds a live feed of detected work, and when the user wants to create content they open the dashboard, pick a work item, and get platform-ready posts for LinkedIn, Twitter, Instagram, YouTube, and Reddit in under thirty seconds. Three agents handle everything: Skippy detects work from connected integrations and builds the content feed, Snooks manages the content calendar and tells the user what to post and when, and Meta writes the actual content in the user's authentic voice across every platform simultaneously. The backend is FastAPI with PostgreSQL, Redis, and Celery. The frontend is Next.js 14 with TypeScript and Tailwind. Everything communicates through a secured REST API and WebSocket layer.

PART ONE — BACKEND
Installation and Environment Setup
Python 3.11 or higher is required. Create a virtual environment with python -m venv venv and activate it. Install all dependencies via pip install -r requirements.txt. The requirements file must contain: fastapi, uvicorn[standard], gunicorn, sqlalchemy[asyncio], asyncpg, alembic, redis, celery[redis], langchain, langchain-anthropic, langchain-google-genai, langchain-openai, anthropic, google-generativeai, openai, pydantic, pydantic-settings, python-jose[cryptography], passlib[bcrypt], python-multipart, httpx, websockets, pillow, python-dotenv, cryptography, and flower. Docker Desktop must be installed for local development. Git must be installed for deployment.
Create a .env file at the project root listed in .gitignore containing: ANTHROPIC_API_KEY, GOOGLE_API_KEY, OPENAI_API_KEY, DATABASE_URL as postgresql+asyncpg://user:password@localhost:5432/cozyjet, REDIS_URL as redis://localhost:6379/0, JWT_SECRET_KEY generated via openssl rand -base64 32, JWT_ALGORITHM as HS256, ACCESS_TOKEN_EXPIRE_MINUTES as 1440, REFRESH_TOKEN_EXPIRE_DAYS as 7, ENCRYPTION_KEY generated via openssl rand -base64 32 for encrypting OAuth tokens at rest, FRONTEND_URL as http://localhost:3000, ALLOWED_ORIGINS as a comma-separated list, GITHUB_CLIENT_ID, GITHUB_CLIENT_SECRET, NOTION_CLIENT_ID, NOTION_CLIENT_SECRET, FIGMA_CLIENT_ID, FIGMA_CLIENT_SECRET, GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, LINKEDIN_CLIENT_ID, LINKEDIN_CLIENT_SECRET, TWITTER_CLIENT_ID, TWITTER_CLIENT_SECRET, and ENVIRONMENT as development or production.

Authentication
The POST /api/auth/signup endpoint accepts email, password, and display_name, validates email format and password minimum eight characters with one uppercase and one number, hashes the password with bcrypt at rounds 12, generates a verification token via secrets.token_urlsafe(32), creates the user, sends a verification email via httpx to a transactional email service, and returns a 201 with the user object and a JWT pair. The POST /api/auth/login endpoint accepts email and password, verifies the hash, checks email_verified is True and returns HTTP 403 with a helpful message if not, generates an access token expiring in 1440 minutes signed with HS256 and a refresh token expiring in 7 days, sets the refresh token in an HttpOnly Secure SameSite=Strict cookie, and returns the access token in the response body. The POST /api/auth/refresh endpoint reads the refresh token from the cookie, validates it, and returns a new access token. The POST /api/auth/logout endpoint blacklists the access token in Redis with TTL equal to remaining token lifetime. The GET /api/auth/verify-email/{token} endpoint sets email_verified True and redirects to the frontend dashboard.
The get_current_user dependency in app/dependencies.py extracts the Authorization Bearer token, checks Redis blacklist, decodes with jose.jwt.decode(), queries the user asynchronously, and returns the user object or raises HTTP 401. This dependency must be injected into every protected route without exception.

Encryption Service
All OAuth tokens must be encrypted before touching the database. The app/services/encryption_service.py module must implement two functions. The encrypt_token(raw_token: str) -> str function uses Python's cryptography.fernet.Fernet initialized with ENCRYPTION_KEY from environment, calls fernet.encrypt(raw_token.encode()), and returns the base64 encoded ciphertext as a string. The decrypt_token(encrypted_token: str) -> str function calls fernet.decrypt(encrypted_token.encode()) and returns the decoded plaintext. These two functions must be called in exactly two places in the entire codebase: encrypt immediately after receiving tokens in the OAuth callback, decrypt immediately before making an outbound API call to the platform. The raw token must never be assigned to any variable outside of these two operations, never logged, and never returned in any API response.

Integration System — OAuth for Every Platform
The GET /api/integrations/connect/{platform} endpoint constructs the OAuth authorization URL for the requested platform and returns it to the frontend which opens it in a popup via window.open(url, 'oauth', 'width=600,height=700'). For GitHub the URL is https://github.com/login/oauth/authorize with client_id, redirect_uri pointing to the callback endpoint, scope as repo read:user, and a random state string stored in Redis with a five-minute TTL. For Notion the URL is https://api.notion.com/v1/oauth/authorize with client_id, response_type as code, and redirect_uri. For Figma it is https://www.figma.com/oauth with client_id, redirect_uri, scope as files:read, and state. For Google Drive and Calendar it is https://accounts.google.com/o/oauth2/v2/auth with client_id, redirect_uri, scope as https://www.googleapis.com/auth/drive.readonly https://www.googleapis.com/auth/calendar.readonly, access_type as offline to receive a refresh token, and state. For LinkedIn it is https://www.linkedin.com/oauth/v2/authorization with client_id, redirect_uri, scope as r_liteprofile w_member_social, and state. For Twitter it uses OAuth 2.0 PKCE with a code verifier and code challenge stored in Redis, authorization URL at https://twitter.com/i/oauth2/authorize with client_id, redirect_uri, scope as tweet.read tweet.write users.read offline.access, code_challenge, and code_challenge_method as S256.
The GET /api/integrations/callback/{platform} endpoint verifies the state parameter against Redis, exchanges the authorization code for tokens via httpx POST to the platform token endpoint, encrypts both tokens using the encryption service, upserts the integrations record with all fields populated, triggers an immediate first sync via skippy_sync_integration.delay(integration_id), publishes an integration_connected event to Redis pub/sub channel ws_user:{user_id}, and returns an HTML page that calls window.opener.postMessage('connected', '*') and then window.close() so the popup closes and the parent window receives the success signal.
The GET /api/integrations/status endpoint returns all integrations for the current user as a list containing platform name, is_active, last_synced_at, platform_username, a preview string generated from the count of seeds detected in the last 24 hours for that platform, and sync_error if present. The DELETE /api/integrations/{platform} endpoint sets is_active to False, attempts token revocation with the platform if supported, deletes the encrypted tokens, and cancels the Celery scheduled sync for that integration.

Skippy Agent and Sync Tasks
The Skippy agent in app/agents/skippy_agent.py is a LangChain chain using Claude Sonnet via langchain-anthropic. The system prompt instructs the agent to act as a work analyst who understands what developers, designers, and creators accomplish in their tools, to read the raw activity data provided, and to return a JSON object with keys title as a concise work title under sixty characters, description as two to three sentences explaining what was done, why it matters professionally, and who would find it interesting on social media, tags as a list of three to five relevant topic strings, and suggested_platforms as a list of platform names best suited for this content. The chain must use ChatPromptTemplate with a system message and a human message that includes the raw activity data as a formatted string.
The Celery tasks in app/tasks/skippy_tasks.py must include skippy_sync_all_users scheduled every two hours that queries all users with at least one active integration and dispatches individual user sync subtasks. The skippy_sync_user task queries all active integrations for the user and dispatches per-integration sync subtasks. The skippy_sync_integration task is the core function that loads the integration, decrypts the access token, makes the platform API call, processes activity items, calls the Skippy LangChain agent for each item, deduplicates against existing seeds by comparing source_url or commit hash in source_metadata, saves new content_seeds records, updates last_synced_at, and publishes new_seed events to Redis for WebSocket delivery.
Platform API calls work as follows. For GitHub: call GET https://api.github.com/user/repos then for each repo call GET https://api.github.com/repos/{owner}/{repo}/commits with since set to two hours ago, extracting commit message, files changed, additions, and deletions. For Notion: call POST https://api.notion.com/v1/search with last_edited_time filter, extracting page titles and parent information. For Figma: call GET https://api.figma.com/v1/me/files sorted by last_modified, processing files updated in the last two hours. For Google Drive: call the Drive API v3 files list with query modifiedTime > '{two_hours_ago}' extracting file names and types. For Google Calendar: call the Calendar API events list with timeMin as start of day and timeMax as now extracting completed event titles and durations.
The on-demand endpoints include POST /api/skippy/sync-now that triggers an immediate sync, POST /api/skippy/voice that accepts a transcription string and creates a seed via the Skippy agent, and POST /api/skippy/screenshot that accepts a multipart image, converts to base64, sends to GPT-4 Vision or Claude with a visual analysis prompt, and creates a seed from the response.

Snooks Agent
The Snooks agent in app/agents/snooks_agent.py uses Google Gemini Pro via langchain-google-genai. The POST /api/snooks/suggest endpoint loads the user's last fourteen days of seeds, last thirty days of analytics, and current top ten trends, passes all context to the Gemini agent requesting five specific weekly content recommendations each with title, angle, platform, best_day, best_time, and reasoning fields, and returns the structured list. The GET /api/snooks/calendar endpoint returns all scheduled and published content for the next thirty days grouped by date. The GET /api/snooks/optimal-times endpoint queries post_analytics grouped by day of week and hour, calculates average engagement per slot per platform, and returns the top three time slots per platform. The GET /api/snooks/trends endpoint returns current trends ordered by score. The Celery detect_trends task runs hourly, queries trending topic sources, and populates the trends table with six-hour expiry on each record.

Meta Agent
The Meta agent in app/agents/meta_agent.py uses Claude Sonnet via langchain-anthropic with ConversationBufferMemory keyed per user ID stored in Redis so memory persists across requests within a twelve-hour session window.
The POST /api/meta/generate endpoint loads the seed by seed_id, loads the user's voice_profile, constructs the LangChain prompt with the seed description as the subject, the full voice profile as style instructions, all platform formatting constraints, and the instruction to generate exactly three variations per requested platform. Variation zero uses emotional storytelling with a hook and narrative arc. Variation one is direct and technical written for professional peers. Variation two leads with outcomes and measurable results. Platform constraints hardcoded into the prompt are: LinkedIn posts 1200 to 1900 characters with three to five hashtags ending in a question or call to action; Twitter threads six to ten tweets each under 280 characters with the first as a strong standalone hook; Instagram captions under 2200 characters conversational with five to ten hashtags at the end; YouTube scripts with an eight-second hook before any introduction and a subscribe prompt at the two-minute mark; Reddit posts detailed and community-first avoiding promotional tone. All generated content saves as individual content records with the appropriate variation_index and returns grouped by platform.
The POST /api/meta/generate-from-idea endpoint accepts a raw topic string and skips seed lookup, passing the topic directly to Meta with the voice profile and thought leadership framing. The POST /api/meta/generate-from-trend endpoint loads a trend by ID and instructs Meta to connect the user's expertise to the trending topic. The POST /api/meta/repurpose endpoint accepts source_text and target_platforms and instructs Meta to transform existing long-form content into native social posts. The POST /api/meta/refine endpoint accepts content_id and instruction, loads the content text and conversation memory, passes both to Meta for revision, saves the updated content, and calls the voice service to log the preference change and update voice_profile. The POST /api/meta/approve endpoint updates status from draft to approved with an optional scheduled_time. The POST /api/meta/chat endpoint accepts message and mode, loads conversation memory, calls the appropriate LangChain chain, and streams the response as Server-Sent Events.

WebSocket Handler
The WS /ws/main?token={jwt} endpoint authenticates by decoding the token query parameter, registers the connection in Redis hash ws_connections:{user_id}, and enters a receive loop handling message types ping responding with pong and chat passing content to the appropriate agent and streaming responses back. A background asyncio task subscribes to Redis pub/sub channel ws_user:{user_id} and forwards all messages from Celery tasks to the connected client. Messages pushed over WebSocket include new_seed when Skippy detects work, integration_connected when OAuth completes, post_published when a scheduled post goes live, sync_complete when a background sync finishes, and trend_alert when a relevant trend is detected. Heartbeats run every thirty seconds with automatic reconnection handling.

Publishing Tasks
The publish_scheduled_content Celery task runs every sixty seconds, queries the content table for approved records where scheduled_time is past, decrypts the user's platform OAuth token, and makes the publishing API call. For LinkedIn it calls POST https://api.linkedin.com/v2/ugcPosts with the content formatted as a share commentary object. For Twitter it calls POST https://api.twitter.com/2/tweets. For Instagram it calls the Facebook Graph API to create a media container then publish it. On success it updates the record status to published and saves the platform_post_id. On failure it saves the error message, sets status to failed, and sends a WebSocket notification. The collect_analytics task runs every six hours, fetches engagement metrics from each platform API for published content in the last thirty days, and upserts into post_analytics.

Docker and Deployment
The docker-compose.yml defines five services. The postgres service uses postgres:15-alpine with a named volume and a health check using pg_isready. The redis service uses redis:7-alpine with --appendonly yes for persistence. The backend service builds from the root Dockerfile, runs gunicorn app.main:app -w 4 -k uvicorn.workers.UvicornWorker -b 0.0.0.0:8000, depends on postgres and redis with condition: service_healthy, and loads all environment variables from .env. The celery_worker service uses the same image with command celery -A app.tasks.celery_app worker --loglevel=info --concurrency=4. The celery_beat service uses the same image with command celery -A app.tasks.celery_app beat --loglevel=info. The Dockerfile uses python:3.11-slim, installs libpq-dev and gcc, copies and installs requirements, copies the application code, and exposes port 8000. For Railway deployment: push to GitHub, connect Railway, add PostgreSQL and Redis plugins which auto-inject connection URLs, create three services for backend, worker, and beat with their respective start commands, and add all API keys and OAuth credentials as environment variables in the Railway dashboard.
