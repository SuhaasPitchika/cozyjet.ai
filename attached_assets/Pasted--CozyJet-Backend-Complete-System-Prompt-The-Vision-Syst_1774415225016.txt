🚀 CozyJet Backend - Complete System Prompt
The Vision & System Architecture
CozyJet is an AI-powered SaaS platform that solves a fundamental problem: digital workers struggle to market their accomplishments. Developers, designers, and creators spend hours building brilliant work but freeze when it's time to promote it on social media. CozyJet changes this by deploying three intelligent AI agents—Skippy, Snooks, and Meta—that automatically transform completed work into viral-ready marketing content while maintaining strict privacy controls. The backend is a production-grade FastAPI application with 4,000+ lines of Python code, featuring 25 RESTful endpoints, WebSocket support for real-time communication, PostgreSQL for data persistence, Redis for caching and rate limiting, Celery for background task processing, and LangChain for AI orchestration. The system is containerized with Docker, deployable to Railway with one click, and built for scalability from day one. The architecture follows clean separation of concerns: the API layer handles HTTP/WebSocket requests, the service layer contains business logic, the integration layer manages AI agents, the task layer processes background jobs, and the data layer ensures persistence and caching. Every component is async/await enabled, fully type-hinted with Pydantic, and secured with JWT authentication, CORS protection, and intelligent rate limiting.
Installation & Dependencies: Setting Up Your Development Environment
Before writing any code, you need to set up your development toolkit. First, install Python 3.11 or higher (the foundation of the backend) using brew install python@3.11 on macOS, sudo apt install python3.11 python3.11-venv python3-pip on Linux, or download from python.org for Windows—Python 3.11 brings 25% performance improvements and better type checking that catches bugs before production. Second, install Git for version control with brew install git on macOS, sudo apt install git on Linux, or download from git-scm.com for Windows—Git enables collaboration, deployment automation, and acts as your time machine for code changes. Third, install Docker and Docker Compose for containerization by downloading Docker Desktop from docker.com for Mac/Windows or running sudo apt install docker.io docker-compose on Linux—Docker packages your entire backend (Python, PostgreSQL, Redis, Celery) into containers that run identically everywhere, solving the "works on my machine" problem forever. Fourth, obtain API keys which are critical for AI functionality: get your Anthropic Claude API key from console.anthropic.com (required for Meta agent, costs ~$3 per million input tokens), get your Google Gemini API key from makersuite.google.com/app/apikey (required for Snooks agent, free tier with 60 requests/minute), and optionally get an OpenAI API key from platform.openai.com (for Skippy's GPT-4 Vision capabilities, falls back to Claude if not provided). For Python dependencies, you'll install via pip install -r requirements.txt which includes: fastapi (web framework), uvicorn and gunicorn (ASGI servers), sqlalchemy and asyncpg (database ORM and PostgreSQL driver), alembic (database migrations), redis and celery (caching and background tasks), langchain, anthropic, google-generativeai, and openai (AI integrations), pydantic and pydantic-settings (data validation), python-jose and passlib (JWT and password hashing), python-multipart (file uploads), and httpx (async HTTP client). The complete setup takes about 15 minutes and provides everything needed for local development, testing, and eventual production deployment.
The Three AI Agents: Architecture, Workflow & Data Flow
Skippy (The Observer Agent) runs as a desktop application with three isolated processes communicating via signed IPC for maximum security: the Screen Observer captures display snapshots every 30 seconds when active (has screen access but zero network access), the Context Analyzer processes screenshots using a local LLM like Ollama to detect work completion signals (has network access but cannot see the screen), and the Chat Interface shows results to users for approval (no screen or analysis capabilities). When Skippy detects completed work—like a Git commit message reading "Add JWT authentication" or a Figma "Saved" notification—it creates a structured work summary including the application, task description, duration estimate, and work type. Users review this summary in the chat interface, approve or reject it, and only approved summaries are sent to the cloud backend via POST /api/skippy/work-events. The backend stores this as a WorkEvent record in PostgreSQL, then immediately sends a WebSocket notification to any connected dashboard so users see "New work detected: JWT Implementation" in real-time. Skippy also provides a chat interface at WS /ws/skippy where users can ask questions like "How do I implement rate limiting?" and receive contextual help based on their current screen, powered by Claude or GPT-4 Vision. Privacy is paramount: users maintain a PII blocklist (stored in their user profile as JSON) of sensitive terms like "salary", "api_key", or "credit_card", and Skippy automatically filters these before sending anything to the cloud—the three-process architecture ensures that screen data, network access, and AI analysis are physically separated, making data exfiltration architecturally impossible.
Snooks (The Productivity Analyst Agent) takes work events from Skippy and transforms them into actionable insights through two primary functions: visual timeline generation and productivity scoring. When a user requests their timeline via POST /api/snooks/timeline with a date range, the backend queries PostgreSQL for all WorkEvent records using the optimized composite index on (user_id, detected_at) for instant retrieval even with millions of records. This data flows to the SnooksTimeline LangChain agent, which formats a prompt for Gemini Pro asking it to analyze work patterns and generate structured visualization data including timeline nodes (individual work sessions with time, title, duration, flow score), Bézier curve threads connecting related work (like multiple commits on the same feature), sticky notes with screenshots and summaries, and an overall narrative summary. Simultaneously, Snooks calculates a productivity score from 0-100 using a sophisticated weighted matrix: focused time (30% weight, measuring uninterrupted deep work hours), flow state quality (30% weight, tracking context switching frequency), distraction frequency (20% weight, counting interruptions), productive ratio (15% weight, comparing active work vs passive scrolling), and tool mastery (5% weight, measuring efficiency with keyboard shortcuts and workflows). Scores above 85 indicate exceptional productivity, 70-85 is good, 50-70 is moderate, and below 50 triggers actionable insights like "You checked email 12 times during deep work—consider batch processing communications." At month-end, POST /api/snooks/wrapped generates a MAXIM Wrapped report—a Three.js animated visualization showing highlights, most productive days, growth trajectory, and shareable cards—powered by Gemini analyzing the entire month's work patterns and generating a narrative arc that makes productivity data emotionally resonant and shareable on social media.
Meta (The Content Strategist Agent) is where work transforms into marketing gold. When users complete work via Skippy and want to share it, they open Meta's interface and call POST /api/meta/generate with the work_event_id and target platform (linkedin/twitter/instagram/youtube/reddit). The backend retrieves the WorkEvent from PostgreSQL, loads the user's voice_profile (JSON containing tone preferences like "professional", style like "conversational", emoji_usage like "moderate", formality level, and humor preferences), and passes everything to the MetaContentGenerator LangChain agent powered by Claude Sonnet 4. Meta doesn't just generate content—it applies platform-specific optimization: LinkedIn posts are 1,300-1,900 characters with professional tone and a hook-insight-value structure, Twitter threads are 6-12 punchy tweets optimized for engagement, Instagram carousels are 7-10 visual slides with compelling copy, and YouTube scripts hook viewers in 8 seconds or they scroll away. Meta generates three variations for every request: a storytelling version that uses narrative hooks and emotional triggers ("Ever wondered how JWT authentication really works? Today I dove deep..."), a technical version for engineer audiences who want specifics ("Implemented JWT auth with refresh tokens. Here's my approach: 1. Token generation using HS256..."), and a results-focused version highlighting outcomes ("Shipped secure authentication in 90 minutes. Zero vulnerabilities. 100% test coverage."). Each variation employs viral mechanics: curiosity gaps that make people keep reading, social proof that builds credibility, specificity that makes claims believable, emotional triggers that make content memorable, and contrast that makes posts stand out. Meta uses ConversationBufferMemory per user, so when you refine content with POST /api/meta/refine, Meta remembers previous versions and your feedback, learning your preferences over time and auto-adjusting the voice_profile. Generated content is saved to the Content table with status='draft', cached in Redis for 1 hour (key: content:generation:{work_event_id}:{platform}), and returned as JSON with all three variations plus posting tips like "Post between 8-10 AM for maximum LinkedIn reach" and "Add an image for 2x engagement."
API Architecture, Data Flow & Frontend Integration
The backend exposes a comprehensive REST API with 25 endpoints organized into five routers: /api/auth handles authentication with 5 endpoints (POST /signup creates users with bcrypt-hashed passwords and sends verification emails, POST /login validates credentials and returns JWT access tokens valid for 24 hours plus refresh tokens valid for 7 days, POST /refresh generates new access tokens from valid refresh tokens, GET /verify-email confirms email addresses via tokens, POST /logout blacklists tokens in Redis to prevent reuse), /api/skippy manages work detection with 5 endpoints (POST /analyze receives screen descriptions and returns work detection results, POST /chat provides contextual assistance, POST /work-events manually creates work records, GET /work-events retrieves paginated history with filters for work_type and date ranges, GET /activity-summary aggregates statistics by period), /api/meta handles content generation with 6 endpoints (POST /generate creates platform-optimized content from work events, POST /refine improves content based on feedback, POST /chat provides strategic content advice, POST /approve marks content ready for publishing and optionally schedules it, GET /content retrieves content history filtered by status and platform, POST /voice-profile updates brand voice preferences), /api/snooks provides analytics with 3 endpoints (POST /timeline generates visual work timelines, POST /productivity-score calculates weighted scores with breakdowns, POST /wrapped creates monthly animated reports), and /ws/ offers 3 WebSocket endpoints (WS /ws/skippy?token=JWT enables real-time chat with streaming responses, WS /ws/notifications?token=JWT pushes live events like "Content published successfully!", WS /ws/stats provides real-time dashboard metrics). Every request flows through a middleware pipeline: CORS middleware validates request origins against ALLOWED_ORIGINS to prevent unauthorized access, rate limiter middleware queries Redis using sliding window algorithm (100 requests/minute for authenticated users, 5/minute for auth endpoints, 10/minute for content generation), authentication middleware extracts JWT tokens from Authorization headers, verifies signatures using JWT_SECRET_KEY, decodes user_id, and attaches it to the request context—invalid or expired tokens return 401 Unauthorized before reaching endpoint code. The frontend (Next.js 14 with TypeScript) integrates via environment variables: NEXT_PUBLIC_API_URL=https://api.cozyjet.com and NEXT_PUBLIC_WS_URL=wss://api.cozyjet.com enable all HTTP and WebSocket communication, with fetch() or axios for REST calls including Authorization: Bearer {access_token} headers, WebSocket connections like new WebSocket('wss://api.cozyjet.com/ws/skippy?token=${accessToken}') for real-time features, TanStack Query for client-side caching and automatic refetching, Zustand for global state management (storing auth tokens, user profile, active work events), and error boundaries catching API failures with automatic retry logic and user-friendly messages—the frontend never accesses the database directly, all data flows through the secured API layer, ensuring proper authentication, validation, and business logic enforcement at every interaction point.
Database Schema, Background Processing & Deployment Pipeline
The PostgreSQL database uses three core tables with optimized relationships: users table stores identity with columns id (UUID primary key), email (unique, indexed), password_hash (bcrypt with 12 rounds), display_name, avatar_url, subscription_tier (enum: free/pro/studio controlling feature access), subscription_expires_at (automatic downgrade when expired), email_verified (boolean blocking premium features until true), verification_token (random string for email confirmation), pii_blocklist (JSON array like ["salary", "api_key"] for Skippy filtering), voice_profile (JSON object with tone, style, emoji_usage for Meta), created_at, and updated_at; work_events table tracks activity with columns id (UUID), user_id (foreign key to users with ON DELETE CASCADE), integration_type (enum: github/figma/notion/gmail/manual), resource_title (e.g., "Fix authentication bug"), work_summary (detailed description), detected_at (timestamp), duration_minutes (actual work time excluding distractions), work_type (enum: coding/design/writing/review/meeting/research), flow_score (0-100 productivity for this session), metadata (JSON for context like repo name, commit hash, files changed), and created_at, with composite indexes idx_work_events_user_detected on (user_id, detected_at DESC) for instant recent work queries and idx_work_events_user_type on (user_id, work_type) for filtered retrieval; content table manages generated posts with columns id (UUID), user_id (foreign key to users ON DELETE CASCADE), work_event_id (nullable foreign key to work_events ON DELETE SET NULL), platform (enum: linkedin/twitter/instagram/youtube/reddit), content_type (enum: post/thread/carousel/video_script/article), content_text (the generated content), status (enum: draft/approved/published/failed driving the publishing workflow), scheduled_time (when to publish, NULL if not scheduled), published_at (when actually published), post_id (platform-specific identifier after publishing), error_message (if publishing failed), created_at, and updated_at, with indexes idx_content_user_status on (user_id, status) for filtered queries and partial index idx_content_scheduled on scheduled_time WHERE scheduled_time IS NOT NULL AND status='approved' optimizing Celery's publishing query. Background processing uses Celery with Redis as the message broker: Celery Beat (scheduler) runs in a dedicated process checking every 60 seconds for scheduled tasks, creating jobs like publish_scheduled_content.apply_async() that get serialized into Redis, and Celery Workers (running in separate processes) poll Redis, deserialize tasks, execute functions (like querying PostgreSQL for approved content where scheduled_time <= now(), calling platform APIs like LinkedIn's ugcPosts endpoint with OAuth tokens stored in the database, updating content records with status='published' and post_id, sending WebSocket notifications), and retry with exponential backoff on failure—scheduled jobs include publish-scheduled-content (every 60s), collect-analytics (hourly), daily-productivity-summary (midnight via crontab), and cleanup-old-data (Sunday 2 AM). Deployment follows a Docker Compose architecture with five services: postgres (PostgreSQL 15 with 5GB storage, automatic backups, DATABASE_URL exposed to other services), redis (Redis 7 with 512MB memory, REDIS_URL exposed, persistence enabled), backend (FastAPI app built from Dockerfile, runs via Gunicorn with 4 UvicornWorker processes, exposes port 8000, depends_on postgres and redis, mounts .env for secrets), celery_worker (same image as backend, custom command celery -A app.tasks.celery_app worker --loglevel=info, shares environment variables, depends_on redis), and celery_beat (same image, command celery -A app.tasks.celery_app beat --loglevel=info, ensures only one scheduler runs)—local development uses docker-compose up -d for instant environment, production deployment to Railway involves pushing code to GitHub, connecting Railway to the repo (auto-detects Dockerfile), adding PostgreSQL and Redis databases (Railway injects DATABASE_URL and REDIS_URL automatically), setting environment variables (ANTHROPIC_API_KEY, GOOGLE_API_KEY, OPENAI_API_KEY, JWT_SECRET_KEY generated via openssl rand -base64 32, FRONTEND_URL, ALLOWED_ORIGINS), deploying three services (backend with Dockerfile, celery_worker with custom start command, celery_beat with custom start command all sharing variables), generating a domain (your-app.up.railway.app), and verifying via curl https://your-app.up.railway.app/health which returns {"status":"healthy","agents":{"skippy":"operational","snooks":"operational","meta":"operational"}}—the entire deployment from code to production takes under 15 minutes with zero-downtime rollouts, automatic SSL certificates, and built-in monitoring, making CozyJet backend production-ready from day one.
