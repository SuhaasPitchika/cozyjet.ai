The Vision & System Overview
CozyJet is an AI-powered SaaS platform that solves a fundamental problem: digital workers spend hours creating brilliant work but struggle to market it on social media. Imagine you just finished building a JWT authentication system after 90 minutes of focused coding. The moment you commit your changes, CozyJet's ecosystem springs to life. Skippy, your desktop AI assistant, observes your screen (with strict privacy controls), detects the completion signal, and understands what you've accomplished. It creates a concise work summary and sends it to the backend. Snooks, the analytics agent, takes this work event and adds it to your visual timeline, calculating a productivity score (0-100) based on five weighted factors: focused time (30%), flow state quality (30%), distraction frequency (20%), productive ratio (15%), and tool mastery (5%). It distinguishes between "3 hours on GitHub" and the reality: "90 minutes of actual focused coding, with 30 minutes in meetings and 45 minutes distracted by notifications." Meta, the content strategist, transforms your work summary into three platform-optimized variations—storytelling, technical, and results-focused—ready for LinkedIn (1300-1900 chars), Twitter (6-12 tweet threads), Instagram (7-10 carousels), or YouTube (scripts with 8-second hooks). You pick your favorite, schedule it for 8 AM tomorrow when your audience is most active, and boom—your coding achievement becomes marketing content without breaking flow. This entire workflow—from work detection to scheduled publishing—runs through a sophisticated backend built on FastAPI, PostgreSQL, Redis, Celery, and LangChain, orchestrating three AI agents (Claude Sonnet 4, Gemini Pro, and optionally GPT-4 Vision) to create a seamless experience.
Required Installations & Setup
Before writing code, install these tools: (1) Python 3.11+ for the core runtime—download from python.org or install via Homebrew on Mac (brew install python@3.11) or apt on Linux (sudo apt install python3.11 python3.11-venv python3-pip). Check installation with python --version. (2) Git for version control and deployment—install via Homebrew (brew install git), apt (sudo apt install git), or download from git-scm.com. Verify with git --version. (3) Docker & Docker Compose for containerization—download Docker Desktop from docker.com for Mac/Windows, or install on Linux with sudo apt install docker.io docker-compose. This lets you run docker-compose up and spin up PostgreSQL, Redis, FastAPI, and Celery workers with one command. Verify with docker --version and docker-compose --version. (4) API Keys from three services: Get Anthropic Claude API key (required for Meta agent) from console.anthropic.com—click "Create Key" and copy the key starting with sk-ant-. This costs $3 per million input tokens and $15 per million output tokens (generating a LinkedIn post costs under 1 cent). Get Google Gemini API key (required for Snooks agent) from makersuite.google.com/app/apikey—sign in with Google, create an API key, and copy it. This has a generous free tier of 60 requests per minute. Optionally get OpenAI API key (for Skippy's GPT-4 Vision) from platform.openai.com—navigate to API Keys and create a new secret key starting with sk-. If you skip this, Skippy falls back to Claude Sonnet. Store these keys safely in a .env file with ANTHROPIC_API_KEY=sk-ant-..., GOOGLE_API_KEY=your-key, OPENAI_API_KEY=sk-... (optional), along with a strong JWT_SECRET_KEY generated via openssl rand -base64 32. These installations take 20 minutes total and give you everything needed to run the backend.
Technology Stack & Architecture
The backend uses FastAPI, a modern Python web framework that's blazing fast (comparable to Node.js) due to native async/await support. Every database query, API call, and Redis operation runs asynchronously—while one request waits for PostgreSQL, another processes. FastAPI auto-generates interactive API docs at /docs (Swagger UI) where you can test every endpoint, and it provides type safety through Pydantic schemas that validate all inputs before your code runs. PostgreSQL 15+ stores all persistent data—users, work events, generated content—with ACID compliance ensuring data consistency under heavy load, advanced indexing for instant queries even with millions of records, and JSON support for flexible fields like user preferences and voice profiles. We use SQLAlchemy 2.0 as the ORM with full async support, so database calls never block. Redis 7+ serves dual purposes: (1) caching frequent queries like productivity scores (TTL 10 minutes) and user profiles (TTL 1 hour) to make APIs feel instant, and (2) rate limiting via sliding window algorithm—users get 100 requests/minute authenticated, 5/minute for auth endpoints, 10/minute for content generation—all tracked in Redis with sub-millisecond checks. Celery handles background tasks: when you schedule content for 8 AM tomorrow, Celery queues it in Redis; Celery Beat (scheduler) checks every 60 seconds and triggers workers to publish content, send emails, or generate analytics. If tasks fail, Celery retries with exponential backoff. LangChain orchestrates the three AI agents—it manages conversation memory for Meta (remembers your previous content and feedback), provides prompt templates so agent behavior can change without code changes, and makes switching models trivial (swap Claude for GPT-4 in one line). The stack is containerized with Docker—the Dockerfile defines the Python environment with all dependencies, and docker-compose.yml orchestrates five services: PostgreSQL, Redis, FastAPI backend, Celery worker, and Celery beat. One command (docker-compose up) starts everything locally.
Complete Data Flow & Agentic Workflow
Step 1: Work Detection. Skippy runs three isolated processes on the user's desktop. The Screen Observer captures displays every 30 seconds but has zero network access. It passes screenshots via signed IPC to the Context Analyzer, which has network access but cannot see the screen—it only receives encrypted screenshot data. The analyzer runs a local LLM (Ollama) to detect completion signals like Git commits or "Saved" indicators. When work completes, it creates a summary shown in the Chat Interface (third process with neither screen nor analysis access). The user reviews and approves, then approved data goes to the cloud. Step 2: API Call. Skippy calls POST /api/skippy/work-events with the work summary. The request hits FastAPI through Gunicorn (multi-process manager). CORS middleware checks the origin is allowed. Rate limiter middleware queries Redis—if the user exceeded 100 requests/minute, returns 429. Auth middleware extracts the JWT token from the Authorization header, verifies it with JWT_SECRET_KEY, decodes the user_id, and attaches it to the request. Step 3: Database Write. The endpoint creates a WorkEvent record via SQLAlchemy, which generates SQL and sends to PostgreSQL. The database validates constraints (user_id exists) and inserts the row, returning the UUID. Simultaneously, the WebSocket ConnectionManager sends a live notification to all the user's active connections: "New work detected: JWT Implementation." Step 4: Content Generation. Later, the user opens Meta and calls POST /api/meta/generate with work_event_id and platform="linkedin". The endpoint fetches the WorkEvent and user's voice_profile from PostgreSQL. LangChain constructs a prompt combining Meta's system instructions, work context, platform requirements (1300-1900 chars), and voice profile (professional tone, moderate emoji usage), then sends to Claude Sonnet 4 via Anthropic API. Claude returns three content variations. The endpoint saves each as a Content record with status="draft" and caches in Redis (key: content:generation:{work_event_id}:linkedin, TTL 1 hour). Step 5: Approval & Scheduling. The user picks a variation and calls POST /api/meta/approve with content_id and scheduled_time="2024-03-24T08:00:00Z". The endpoint updates the Content record: status="approved", scheduled_time set. Step 6: Background Publishing. Celery Beat checks every 60 seconds for approved content where scheduled_time <= now(). At 8:00 AM, it creates a publish_scheduled_content task serialized to Redis. A Celery Worker polls Redis, picks up the task, calls the LinkedIn API with the user's OAuth token, receives a post_id, and updates the Content record: status="published", published_at=now(), post_id saved. If the API call fails, it updates status="failed" with error_message and Celery retries with exponential backoff. The worker sends a WebSocket notification: "Content published successfully!" Step 7: Productivity Analysis. When the user opens Snooks and calls POST /api/snooks/timeline, the endpoint queries PostgreSQL for all WorkEvents in the date range using the composite index (user_id, detected_at DESC) for instant retrieval. This data goes to the SnooksTimeline LangChain agent, which prompts Gemini Pro to generate timeline visualization data (nodes, threads, summary). Simultaneously, it calculates productivity score by aggregating flow_scores, counting distractions, measuring focused time, and applying the weighted formula. The response combines timeline + score and caches in Redis (TTL 10 minutes). This entire flow—from Skippy detecting work to Celery publishing content to Snooks generating analytics—showcases the agentic workflow where three AI agents collaborate through a sophisticated backend orchestrating databases, caches, background workers, and real-time WebSocket updates.
Database Schema & API Structure
Three core tables: (1) Users with id (UUID primary key), email (unique), password_hash (bcrypt 12 rounds), subscription_tier (enum: free/pro/studio), pii_blocklist (JSON array of blocked keywords like "salary" or "api_key"), and voice_profile (JSON with tone, style, emoji_usage for brand consistency). Foreign keys use ON DELETE CASCADE so deleting a user cascades to all their data. (2) WorkEvents with user_id (FK to users), integration_type (github/figma/notion/gmail/manual), resource_title ("Fix authentication bug"), work_summary (detailed description), detected_at (timestamp), duration_minutes (actual work time excluding distractions), work_type (enum: coding/design/writing/review/meeting/research), flow_score (0-100), and metadata (JSON with extra context like repo name, commit hash). Indexed via (user_id, detected_at DESC) for instant recent event retrieval and (user_id, work_type) for filtering. (3) Content with user_id (FK), work_event_id (FK, nullable, ON DELETE SET NULL), platform (linkedin/twitter/instagram/youtube/reddit), content_text (the generated post), status (draft/approved/published/failed), scheduled_time (when to publish), published_at (when actually published), post_id (platform's returned ID), and error_message (if failed). Indexed via (user_id, status) for fetching drafts/published content and a partial index on scheduled_time WHERE status="approved" for Celery's publishing queries. Relationships: User → WorkEvents (one-to-many), User → Content (one-to-many), WorkEvent → Content (one-to-one or one-to-many). The API exposes 25 endpoints: 5 auth endpoints (signup, login, refresh, verify-email, logout with JWT), 5 Skippy endpoints (analyze, chat, work-events POST/GET, activity-summary), 6 Meta endpoints (generate, refine, chat, approve, content history, voice-profile), 3 Snooks endpoints (timeline, productivity-score, wrapped), 3 WebSocket endpoints (skippy chat streaming, notifications, stats), and 3 system endpoints (root, health, info). Authentication uses JWT with HS256—access tokens expire in 24 hours, refresh tokens in 7 days. Rate limiting is Redis-based with sliding windows. All endpoints return JSON with proper HTTP status codes (201 for creation, 401 for auth failures, 429 for rate limits). The architecture supports real-time via WebSocket (ConnectionManager tracks all active connections and broadcasts to specific users), background processing via Celery (publish content, send emails, generate wrapped reports), and intelligent caching via Redis (user profiles, productivity scores, content generations). Deployment is one-click to Railway—push to GitHub, connect Railway, add PostgreSQL + Redis services, set environment variables (DATABASE_URL, REDIS_URL, API keys, JWT_SECRET_KEY), and Railway auto-detects the Dockerfile and deploys. Add Celery worker and beat as separate services with custom start commands. The entire backend—4,000+ lines of production code—is designed for rapid iteration: async/await for performance, Docker for reproducibility, type hints for safety, and LangChain for AI flexibility. It's ready to handle thousands of concurrent users, process millions of work events, generate unlimited content, and scale horizontally by adding more workers.
Quick Start & Deployment
Local development: Extract the backend, run cp .env.example .env, edit .env with your API keys (ANTHROPIC_API_KEY=sk-ant-..., GOOGLE_API_KEY=..., JWT_SECRET_KEY=<32+ char secret>), then run docker-compose up -d. This starts PostgreSQL on port 5432, Redis on 6379, FastAPI on 8000, Celery worker, and Celery beat. Visit http://localhost:8000/docs for interactive API documentation where you can test every endpoint. Create a user via POST /api/auth/signup, login to get JWT tokens, then use the token in Authorization headers for authenticated endpoints. Production deployment to Railway: Push code to GitHub, go to railway.app, create a new project from your GitHub repo, add PostgreSQL and Redis databases (Railway auto-configures DATABASE_URL and REDIS_URL), add environment variables (ANTHROPIC_API_KEY, GOOGLE_API_KEY, JWT_SECRET_KEY, FRONTEND_URL, ALLOWED_ORIGINS), then add two additional services: Celery worker with start command celery -A app.tasks.celery_app worker --loglevel=info and Celery beat with celery -A app.tasks.celery_app beat --loglevel=info. Railway generates a domain like your-app.up.railway.app, deploys automatically on every Git push, handles SSL certificates, and provides monitoring/logs. The entire backend—from local setup to production deployment—takes under an hour. For vibe coding: Clone the repo, read the architectural overview to understand data flow, follow the installation guide for tools/API keys, run Docker Compose locally, open the Swagger docs at /docs to explore endpoints interactively, and start building features incrementally—create endpoints, wire up agents, test with Postman, iterate rapidly. The codebase is organized into clear modules (models, schemas, api, services, tasks) so you can work on one feature without touching others. SQLAlchemy handles database migrations, Pydantic validates inputs, LangChain abstracts AI complexity, and FastAPI's automatic docs make testing effortless. This is a production-ready SaaS backend designed for builders who want to ship fast, scale easily, and focus on features rather than infrastructure.
