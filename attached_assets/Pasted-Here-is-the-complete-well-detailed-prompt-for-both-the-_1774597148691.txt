CozyJet — Complete Backend and Frontend Dashboard Build Prompt

PART ONE: BACKEND
Foundation, Installation, and Project Setup
The CozyJet backend is a production-grade Python application built on FastAPI with async PostgreSQL, Redis, Celery, and LangChain powering three AI agents. Before writing any code the following must be installed and configured on the development machine. Python 3.11 or higher is required — install via brew install python@3.11 on macOS or sudo apt install python3.11 python3.11-venv on Ubuntu. Docker Desktop must be installed from docker.com to run the full stack locally. Git must be installed for version control and deployment triggers.
Create the project directory, initialize a virtual environment with python -m venv venv, activate it, and install all dependencies via pip install -r requirements.txt. The requirements file must contain exactly these packages: fastapi, uvicorn[standard], gunicorn, sqlalchemy[asyncio], asyncpg, alembic, redis, celery[redis], langchain, langchain-anthropic, langchain-google-genai, langchain-openai, anthropic, google-generativeai, openai, pydantic, pydantic-settings, python-jose[cryptography], passlib[bcrypt], python-multipart, httpx, websockets, pillow, python-dotenv, cryptography, and flower for Celery monitoring.
Create a .env file in the project root that is listed in .gitignore and must never be committed. This file contains: ANTHROPIC_API_KEY from console.anthropic.com, GOOGLE_API_KEY from makersuite.google.com, OPENAI_API_KEY from platform.openai.com, DATABASE_URL in the format postgresql+asyncpg://user:password@localhost:5432/cozyjet, REDIS_URL as redis://localhost:6379/0, JWT_SECRET_KEY generated via openssl rand -base64 32, JWT_ALGORITHM as HS256, ACCESS_TOKEN_EXPIRE_MINUTES as 1440, REFRESH_TOKEN_EXPIRE_DAYS as 7, FRONTEND_URL as http://localhost:3000 in development, ALLOWED_ORIGINS as a comma-separated list of allowed frontend URLs, ENCRYPTION_KEY generated via openssl rand -base64 32 for encrypting OAuth tokens at rest, and ENVIRONMENT as development or production.
The project folder structure must be organized as follows. The root contains docker-compose.yml, Dockerfile, requirements.txt, .env, .gitignore, and alembic.ini. Inside the app/ directory: main.py is the FastAPI application entry point, config.py holds all Pydantic settings reading from the .env file, database.py manages the async SQLAlchemy engine and session factory, dependencies.py holds all FastAPI dependency injection functions including JWT authentication. The app/models/ directory contains user.py, integration.py, content_seed.py, content.py, analytics.py, tune.py, and __init__.py importing all models. The app/schemas/ directory contains matching Pydantic schemas for every model. The app/api/ directory contains auth.py, integrations.py, skippy.py, snooks.py, meta.py, analytics.py, tune.py, and websockets.py as route modules. The app/agents/ directory contains skippy_agent.py, snooks_agent.py, and meta_agent.py as LangChain agent definitions. The app/services/ directory contains integration_service.py, content_service.py, voice_service.py, and encryption_service.py as business logic layers. The app/tasks/ directory contains celery_app.py, skippy_tasks.py, snooks_tasks.py, publishing_tasks.py, and analytics_tasks.py.

Database Models — Every Table in Full Detail
The users table must be defined in app/models/user.py using SQLAlchemy async ORM with the following columns: id as UUID primary key with server default gen_random_uuid(), email as String unique and indexed, password_hash as String storing bcrypt output at cost factor 12, display_name as nullable String, avatar_url as nullable String, subscription_tier as SQLAlchemy Enum with values free, pro, studio defaulting to free, subscription_expires_at as nullable DateTime, email_verified as Boolean defaulting to False, verification_token as nullable String, voice_profile as JSON column defaulting to an empty dict that stores keys tone (string), style (string), emoji_usage (string: none, low, medium, high), formality (integer 1 through 10), humor (integer 1 through 10), technical_depth (integer 1 through 10), preferred_platforms (list of strings), and learned_preferences (list of preference change log strings), timezone as String defaulting to UTC, created_at and updated_at as DateTime columns with automatic management via SQLAlchemy onupdate.
The integrations table must be defined in app/models/integration.py with columns: id as UUID primary key, user_id as UUID foreign key referencing users.id with cascade delete, platform as Enum with values github, notion, figma, google_drive, google_calendar, linkedin, twitter, instagram, access_token as Text storing the AES-256 encrypted OAuth access token (never plaintext), refresh_token as nullable Text storing the encrypted refresh token, token_expires_at as nullable DateTime, platform_user_id as nullable String storing the user's ID on that platform, platform_username as nullable String, scopes as JSON storing the list of granted OAuth scopes, is_active as Boolean defaulting to True, last_synced_at as nullable DateTime, sync_error as nullable Text storing the last error message if sync failed, and created_at as DateTime.
The content_seeds table must be defined in app/models/content_seed.py with columns: id as UUID primary key, user_id as UUID foreign key with cascade delete, title as String up to 255 characters, description as Text containing the AI-generated summary of the work, source_platform as String (github, notion, figma, google_drive, google_calendar, voice, screenshot, manual), source_url as nullable String, source_metadata as JSON storing platform-specific raw data like commit hash, file ID, or page ID, tags as a PostgreSQL ARRAY of String for categorization, is_used as Boolean defaulting to False tracking whether content has been generated from this seed, is_dismissed as Boolean defaulting to False, and created_at as DateTime. A composite index on (user_id, created_at DESC) is required for fast feed loading.
The content table must be defined in app/models/content.py with columns: id as UUID primary key, user_id as UUID foreign key with cascade delete, seed_id as nullable UUID foreign key referencing content_seeds.id set null on delete, platform as Enum (linkedin, twitter, instagram, youtube, reddit), content_type as Enum (post, thread, carousel, video_script, article), content_text as Text, variation_index as Integer (0, 1, or 2 representing the three generated variations), status as Enum (draft, approved, scheduled, published, failed) defaulting to draft, scheduled_time as nullable DateTime, published_at as nullable DateTime, platform_post_id as nullable String, error_message as nullable Text, created_at and updated_at as DateTimes. Required indexes: composite on (user_id, status), partial on scheduled_time WHERE scheduled_time IS NOT NULL AND status = 'approved' for Celery publishing query performance.
The post_analytics table stores engagement data fetched from social platforms with columns: id as UUID, content_id as UUID foreign key referencing content.id with cascade delete, platform as String, views as Integer defaulting to 0, likes as Integer defaulting to 0, comments as Integer defaulting to 0, shares as Integer defaulting to 0, engagement_rate as Float calculated as (likes + comments + shares) / max(views, 1) * 100, and fetched_at as DateTime.
The trends table stores trending topics populated by hourly Celery tasks with columns: id as UUID, topic as String, platform as String, trend_score as Integer 0 through 100, related_keywords as JSON array, niche_tags as JSON array for matching to user niches, detected_at as DateTime, and expires_at as DateTime. An index on (platform, trend_score DESC) is required.
The tune_samples table stores the writing samples users submit for voice analysis with columns: id as UUID, user_id as UUID foreign key with cascade delete, sample_text as Text, platform as nullable String, source as Enum (manual, imported, approved_post), and created_at as DateTime.
The chat_sessions table stores conversation history for the Meta chat interface with columns: id as UUID, user_id as UUID foreign key, mode as Enum (refine, meta, support), messages as JSON array where each message is an object with role (user or assistant) and content string, content_id as nullable UUID foreign key for refine sessions linking to the content being edited, and created_at and updated_at as DateTimes.

Authentication System
Authentication must be implemented in app/api/auth.py using JWT with bcrypt password hashing. The POST /api/auth/signup endpoint accepts a JSON body with email, password, and display_name, validates the email format and password strength (minimum eight characters, one uppercase, one number), hashes the password with passlib.hash.bcrypt at rounds 12, generates a random verification token using secrets.token_urlsafe(32), creates the user record, sends a verification email via httpx to a transactional email service, and returns a 201 response with the user object and a JWT pair. The POST /api/auth/login endpoint accepts email and password, retrieves the user, verifies the password hash, checks email_verified is True (return 403 with a helpful message if not), generates an access token expiring in 1440 minutes and a refresh token expiring in 7 days using python-jose, sets the refresh token in an HttpOnly Secure SameSite=Strict cookie, and returns the access token in the response body. The POST /api/auth/refresh endpoint reads the refresh token from the cookie, validates it, generates a new access token, and returns it. The POST /api/auth/logout endpoint blacklists the current access token by storing it in Redis with a TTL equal to the remaining token lifetime. The GET /api/auth/verify-email/{token} endpoint finds the user by verification token, sets email_verified to True, clears the token, and redirects to the frontend dashboard.
The get_current_user dependency in app/dependencies.py must extract the Authorization Bearer token from the request header, check Redis to ensure it is not blacklisted, decode it with jose.jwt.decode(), extract user_id, query the database asynchronously for the user, and return the user object or raise HTTPException(status_code=401). This dependency must be injected into every protected route.

Integration System — OAuth Flows for Every App
The integration system is implemented in app/api/integrations.py and app/services/integration_service.py. All OAuth tokens stored in the integrations table must be encrypted using AES-256-GCM via the cryptography library before saving and decrypted only in memory when making API calls — the raw tokens must never be logged or returned to the frontend.
The GET /api/integrations/connect/{platform} endpoint generates the OAuth authorization URL for the requested platform and returns it to the frontend which opens it in a popup window. For GitHub the authorization URL uses https://github.com/login/oauth/authorize with scopes repo read:user. For Notion it uses https://api.notion.com/v1/oauth/authorize with the integration client ID. For Figma it uses https://www.figma.com/oauth with scopes files:read. For Google Drive and Calendar it uses https://accounts.google.com/o/oauth2/v2/auth with scopes https://www.googleapis.com/auth/drive.readonly and https://www.googleapis.com/auth/calendar.readonly. For LinkedIn it uses https://www.linkedin.com/oauth/v2/authorization with scopes r_liteprofile w_member_social. For Twitter it uses OAuth 2.0 PKCE flow at https://twitter.com/i/oauth2/authorize with scopes tweet.read tweet.write users.read offline.access.
The GET /api/integrations/callback/{platform} endpoint receives the authorization code from the OAuth provider, exchanges it for tokens via httpx POST to the provider's token endpoint, encrypts the tokens using the encryption service, saves or updates the integrations record, triggers an immediate first Skippy sync via skippy_sync_integration.delay(integration_id) as a Celery task, and returns a success response that the frontend popup window can detect to close itself and notify the parent window.
The GET /api/integrations/status endpoint returns all integrations for the current user as a list of objects containing platform name, is_active boolean, last_synced_at timestamp, platform_username, and a preview string like "3 commits detected yesterday" generated from the most recent seeds for that platform.
The DELETE /api/integrations/{platform} endpoint sets the integration is_active to False, revokes the token with the OAuth provider via httpx if the provider supports revocation, and deletes the encrypted tokens from the database.

Skippy Agent — Background Data Connector
The Skippy agent is defined in app/agents/skippy_agent.py as a LangChain chain using Claude Sonnet via langchain-anthropic. The chain takes raw work data as input and returns a structured content seed description. The system prompt must instruct the agent to act as a work analyst who understands what developers, designers, and creators accomplish in their daily tools, to summarize the detected work in two to three sentences that explain what was done, why it matters professionally, and what audience it would resonate with on social media, and to always output valid JSON with keys title, description, tags as a list of strings, and suggested_platforms as a list of strings.
The Celery tasks in app/tasks/skippy_tasks.py must include the following. The skippy_sync_all_users task runs every two hours via Celery Beat and queries all users with at least one active integration, then for each user calls skippy_sync_user.delay(user_id) as a subtask. The skippy_sync_user task queries all active integrations for the given user and for each calls skippy_sync_integration.delay(integration_id). The skippy_sync_integration task is the core sync function — it loads the integration record, decrypts the access token, makes the appropriate API call based on platform using httpx, processes the response to extract recent activity, calls the Skippy LangChain agent for each activity item to generate a seed description, saves new content_seeds records avoiding duplicates by checking source_url or source_metadata against existing records, updates last_synced_at on the integration, and sends a WebSocket notification to the user via Redis pub/sub so the dashboard feed updates in real time.
The platform-specific API calls must work as follows. For GitHub: call GET https://api.github.com/user/repos with the user's token to get repos, then for each active repo call GET https://api.github.com/repos/{owner}/{repo}/commits?since={two_hours_ago} to get recent commits, extracting the commit message, files changed count, and additions and deletions stats. For Notion: call POST https://api.notion.com/v1/search with filter last_edited_time greater than two hours ago to get recently edited pages and databases, extracting page titles and last edited time. For Figma: call GET https://api.figma.com/v1/me/files to get recent files sorted by last modified. For Google Drive: call the Drive API v3 files list endpoint with query modifiedTime > '{two_hours_ago}' and fields including name, mimeType, modifiedTime, and webViewLink. For Google Calendar: call the Calendar API events list for the primary calendar with timeMin as start of today and timeMax as now to get completed events, filtering out declined events.
The on-demand endpoints at POST /api/skippy/sync-now trigger an immediate sync outside the scheduled window. The POST /api/skippy/voice endpoint accepts a JSON body with transcription as a string, passes it to the Skippy LangChain agent with additional context that this is a user-described work item rather than integration-detected data, and returns the created seed. The POST /api/skippy/screenshot endpoint accepts a multipart file upload, converts the image to base64, sends it to GPT-4 Vision or Claude with a prompt asking it to identify visible work and produce a seed description, and returns the created seed.

Snooks Agent — Content Strategy and Calendar
The Snooks agent is defined in app/agents/snooks_agent.py using Google Gemini Pro via langchain-google-genai. The system prompt instructs the agent to act as a senior content strategist who understands platform algorithms, audience psychology, and viral content mechanics, to analyze the provided data and return structured JSON recommendations.
The POST /api/snooks/suggest endpoint loads the authenticated user's last fourteen days of content_seeds, their last thirty days of post_analytics to understand what performs best, and the current top ten trends from the trends table filtered by the user's preferred platforms, passes all of this context to the Snooks Gemini agent with a prompt requesting five specific content ideas for the current week each with a title, angle, platform, best_day, best_time, and reasoning field, and returns the structured suggestions.
The GET /api/snooks/calendar endpoint queries content records joined with content_seeds for the authenticated user where status is scheduled or published in the next thirty days and returns them formatted as calendar objects grouped by date.
The GET /api/snooks/optimal-times endpoint queries post_analytics for the user grouped by EXTRACT(DOW FROM fetched_at) and EXTRACT(HOUR FROM fetched_at), calculates average engagement rate per slot per platform, and returns the top three time slots per platform as objects with day_of_week, hour, and avg_engagement_rate.
The GET /api/snooks/trends endpoint queries the trends table for records where expires_at is in the future ordered by trend_score DESC limited to twenty results, and returns them with all fields. This data is populated by the detect_trends Celery task that runs every hour and calls Google Trends RSS feeds and Twitter trending API to find viral topics in the technology, design, and entrepreneurship niches.

Meta Agent — AI Copywriter and Content Factory
The Meta agent is defined in app/agents/meta_agent.py using Claude Sonnet via langchain-anthropic with ConversationBufferMemory initialized per user ID and stored in Redis so memory persists across API requests within a session. When a new session starts the memory is loaded from Redis; when a session ends or after twelve hours the memory is cleared.
The POST /api/meta/generate endpoint is the primary generation route. It loads the content_seeds record by the provided seed_id, loads the user's voice_profile from the users table, constructs the Meta LangChain prompt that includes the seed description as the work to write about, the full voice profile as writing style instructions, the platform constraints for each requested platform, and the instruction to generate exactly three variations per platform where variation zero is emotional storytelling with a hook and narrative arc, variation one is direct and technical written for professional peers, and variation two leads with outcomes and measurable results. The chain runs and returns structured JSON which is parsed and saved as individual content records with the appropriate variation_index and platform fields, all with status draft. The endpoint returns all created content records grouped by platform.
Platform constraints that must be hardcoded into the Meta prompt are: LinkedIn posts must be 1,200 to 1,900 characters, professional yet personable tone, three to five relevant hashtags, end with a question or call to action; Twitter threads must be six to ten tweets each strictly under 280 characters, first tweet is a strong standalone hook, last tweet ends with a follow call to action, number each tweet; Instagram captions must be under 2,200 characters, conversational and visual-first, use line breaks for readability, five to ten hashtags at the end; YouTube scripts must open with an eight-second hook before any introduction, include a subscribe reminder around the two-minute mark, structure in three clear sections, end with a community question; Reddit posts must be detailed and community-first, avoid sounding promotional, lead with the value or learning rather than the personal achievement.
The POST /api/meta/generate-from-idea endpoint accepts a topic string and platforms array, skips seed lookup, and passes the topic directly to Meta with the voice profile and an instruction that the user wants to write thought leadership content on this topic even without recent related work.
The POST /api/meta/generate-from-trend endpoint accepts a trend_id, loads the trend from the trends table, and instructs Meta to generate content that connects the user's expertise (summarized from their last five seeds) to the trending topic for maximum discoverability.
The POST /api/meta/repurpose endpoint accepts source_text as a long-form string (blog post, transcript, newsletter) and a target_platforms array, and instructs Meta to transform the content into native social posts for each platform while preserving the key ideas and the user's voice.
The POST /api/meta/refine endpoint accepts a content_id and instruction string, loads the existing content text and the user's conversation memory from Redis, passes both to Meta asking it to revise the content according to the instruction while maintaining the established voice profile, saves the revised text as an updated content record, and calls the voice service to log the preference change. The voice service compares the original and revised content to detect patterns — shorter sentences, removed jargon, added emojis, changed opening structure — and updates the relevant fields in the user's voice_profile JSON, appending a human-readable log entry to the learned_preferences array.
The POST /api/meta/approve endpoint updates a content record's status from draft to approved and optionally accepts a scheduled_time DateTime to set scheduling.
The POST /api/meta/chat endpoint accepts a message string and mode string (meta or support), loads the user's conversation memory from Redis, passes the message to the appropriate LangChain chain (Meta agent for content help, a support-instructed Claude chain for product help), streams the response back as Server-Sent Events so the frontend can render tokens progressively, and saves the exchange to the chat_sessions table.

WebSocket Handler
The WebSocket endpoint is implemented in app/api/websockets.py. The WS /ws/main?token={jwt} endpoint authenticates the connection by decoding the token query parameter, registers the connection in a Redis hash keyed ws_connections:{user_id} storing the connection object reference, enters a receive loop listening for incoming messages, and handles two message types: ping responds with pong to maintain the heartbeat, and chat passes the message content to the appropriate LangChain agent and streams responses back as a series of WebSocket messages with a streaming flag and a final message with done flag. A Redis pub/sub subscriber runs in a background asyncio task within the WebSocket handler and listens on the channel ws_user:{user_id} — when Celery tasks complete (new seed detected, post published, sync finished) they publish to this Redis channel and the WebSocket handler forwards the message to the connected client. Connections must implement ping/pong heartbeats every thirty seconds and gracefully handle disconnection by removing the connection from the Redis registry.

Celery Background Tasks
The Celery application in app/tasks/celery_app.py must be configured with Redis as broker and backend, and Celery Beat must define the following scheduled tasks. skippy_sync_all_users runs every two hours. detect_trends runs every hour and queries Google Trends RSS and Twitter trending to populate the trends table. publish_scheduled_content runs every sixty seconds and queries the content table for approved records where scheduled_time is past, attempts publishing via the appropriate platform API using the user's stored OAuth token, updates the record status, and publishes a WebSocket notification. collect_analytics runs every six hours and queries published content from the last thirty days, fetches current engagement from each platform's API, and upserts into post_analytics. send_daily_digest runs every morning at eight AM per user timezone and sends an email summarizing yesterday's performance and today's content suggestions from Snooks.

Docker and Deployment
The docker-compose.yml must define five services: postgres using postgres:15-alpine with a named volume and health check, redis using redis:7-alpine with --appendonly yes for persistence, backend built from the root Dockerfile running gunicorn app.main:app -w 4 -k uvicorn.workers.UvicornWorker -b 0.0.0.0:8000 with all environment variables from .env, celery_worker using the same image with command celery -A app.tasks.celery_app worker --loglevel=info --concurrency=4, and celery_beat using the same image with command celery -A app.tasks.celery_app beat --loglevel=info. The Dockerfile must use python:3.11-slim, install system dependencies including libpq-dev and gcc, copy and install requirements, copy the application, and expose port 8000. For Railway deployment push to GitHub, connect Railway, add PostgreSQL and Redis plugins which auto-inject their URLs, create three services for backend, worker, and beat with their respective start commands, and set all API key environment variables in the Railway dashboard.

PART TWO: FRONTEND DASHBOARD
Stack and Global Setup
The frontend is built with Next.js 14 using the App Router, TypeScript strict mode, Tailwind CSS with a custom design system, Framer Motion for all animations, Radix UI for accessible component primitives, TanStack Query v5 for server state management, Zustand for client state, and Axios for all HTTP requests. Install via npx create-next-app@latest cozyjet-frontend --typescript --tailwind --app then add npm install @tanstack/react-query framer-motion @radix-ui/react-dialog @radix-ui/react-tabs @radix-ui/react-dropdown-menu @radix-ui/react-switch @radix-ui/react-tooltip zustand axios recharts date-fns.
The Tailwind config must extend the default theme with a custom color palette: brand as a scale from 50 to 950 using indigo-adjacent deep blue, surface as neutral grays for card backgrounds, and semantic colors for success, warning, and error. Custom border radius values and a custom font family using Inter from Google Fonts must be configured. Custom animation utilities for fade-up, fade-in, and slide-in must be defined using Tailwind's keyframes extension.
The Zustand store in lib/store.ts must define the following state slices: auth store with user object, accessToken string, setUser, setAccessToken, and logout actions with the token persisted to localStorage via the Zustand persist middleware; UI store with sidebarCollapsed boolean, activeSeedId string or null, notifications array, addNotification, and clearNotification actions; and WebSocket store with isConnected boolean, setConnected, and lastMessage object.
The Axios instance in lib/api.ts must set baseURL to process.env.NEXT_PUBLIC_API_URL, add a request interceptor that reads the access token from the Zustand auth store and attaches it as Authorization: Bearer {token}, and add a response interceptor that catches 401 errors and calls POST /api/auth/refresh silently before retrying the original request using the new token, or calls the Zustand logout action and redirects to /login if refresh fails.
The WebSocket context in lib/websocket-context.tsx must be a React context provider wrapping the entire dashboard layout that on mount establishes a WebSocket connection to NEXT_PUBLIC_WS_URL/ws/main?token={accessToken}, sets up a thirty-second heartbeat interval sending {type: "ping"} messages, listens for incoming messages and routes them to the appropriate handler (new seed notifications update the TanStack Query cache for the seeds list, post published notifications add to the Zustand notifications array, sync complete notifications update the integrations status), handles disconnection by attempting reconnection with exponential backoff starting at two seconds and doubling up to sixty seconds maximum, and provides sendMessage and isConnected via context to all child components.
All TanStack Query hooks must be defined in the hooks/ directory. useContentSeeds(filters) calls GET /api/skippy/seeds with filter params and has a stale time of thirty seconds. useGenerateContent() is a mutation calling POST /api/meta/generate. useCalendar(month, year) calls GET /api/snooks/calendar. useTrends() calls GET /api/snooks/trends with a two-minute stale time. useAnalytics(period) calls GET /api/analytics/summary. useIntegrations() calls GET /api/integrations/status. useVoiceProfile() calls GET /api/tune/voice-profile. All mutations must implement optimistic updates where appropriate so the UI responds instantly without waiting for server confirmation.

Global Chat Interface
A floating chat button must be present on every dashboard page rendered in the root dashboard layout. The button is a circular 56 pixel button with the CozyJet brand color and a chat bubble icon, fixed to the bottom right corner with a subtle shadow and a gentle pulse animation. Clicking it opens a slide-up chat panel that is 400 pixels wide and 560 pixels tall, anchored to the bottom right, with a smooth Framer Motion spring animation. The panel header shows "Meta — Your Marketing AI" with an online indicator dot and a close button. The message list renders user messages right-aligned with the brand color background and assistant messages left-aligned with a light gray background, with the assistant avatar in the top left corner of each assistant bubble. Message text supports markdown rendering for bold, code blocks, and bullet lists. At the bottom an input field with a send button allows typing, with Enter to send and Shift+Enter for newline. Messages are sent via POST /api/meta/chat with mode meta and responses stream back via the WebSocket connection rendering progressively. Suggested prompts appear above the input when the conversation is empty: "What should I post about today?", "Write me a LinkedIn post about my recent work", "Give me five content ideas for this week", and "Help me grow my Twitter following." This chat must have full memory of the conversation within the session powered by the Redis-backed ConversationBufferMemory.
