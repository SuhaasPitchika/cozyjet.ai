# MAXIM AGENT ARCHITECTURE & SECURITY IMPLEMENTATION

## The Complete Technical Specification for Unhackable Local-First AI Agents with Advanced Intelligence

This document provides exhaustive technical detail on MAXIM's revolutionary three-agent architecture featuring Skippy the seeing and understanding agent, Flippo the productivity visualization and timeline agent, and Snooks the viral content strategist. The specification covers the security implementation that makes the system genuinely unhackable, the workflow orchestration that coordinates agent activities, and the backend infrastructure that supports distributed intelligence while preserving absolute privacy. Every paragraph contains implementation-critical information that must be understood before writing code. This specification assumes you are building a system where user trust is the primary asset and any security compromise destroys the entire value proposition. The architecture described here implements defense-in-depth where compromising one component does not cascade into system-wide breach, zero-trust principles where no component trusts any other component without cryptographic verification, and fail-secure defaults where errors result in locked-down states rather than permissive states.

## THE FUNDAMENTAL ARCHITECTURE PHILOSOPHY AND THREAT MODEL

MAXIM's architecture begins with a threat model that assumes sophisticated adversaries with significant resources attempting to extract user data, inject malicious code, or compromise the integrity of generated content. The adversaries include nation-state actors targeting specific high-value users, organized cybercriminals seeking to monetize stolen data through identity theft or corporate espionage, malicious insiders with access to MAXIM's infrastructure attempting to abuse their privileges, supply chain attackers who compromise dependencies or build processes to inject backdoors, and opportunistic attackers scanning for common vulnerabilities. This threat model shapes every architectural decision because underestimating adversary capabilities leads to false security assumptions that collapse under real attack. The architecture implements multiple independent security mechanisms so that defeating one mechanism does not provide system access, each mechanism is simple enough to audit thoroughly preventing subtle implementation bugs, and all mechanisms assume compromise of adjacent components meaning security never depends on trusting another component.

The fundamental architectural principle is local-first processing where user data lives on the user's device and computation happens locally whenever possible, with cloud services used only for coordination and approved output delivery. This principle means that even if attackers completely compromise MAXIM's cloud infrastructure including all servers, databases, and administrative access, they cannot retroactively access user workspace data because that data never existed in the cloud. The architecture uses end-to-end encryption where data leaving the user's device is encrypted with keys that only that device possesses, meaning the cloud backend stores only ciphertext that is useless without the decryption key. The system implements client-side verification where the desktop client cryptographically verifies the identity and integrity of the cloud backend before communicating, preventing man-in-the-middle attacks where adversaries impersonate legitimate servers. The architecture enforces minimal data collection where the cloud backend receives only the absolute minimum information necessary for functionality, with all optional data collection disabled by default and requiring explicit opt-in.

The three-agent architecture distributes different responsibilities to different agents not just for user comprehension but for security isolation and cognitive clarity. Skippy is the seeing and understanding agent who acts as an intelligent virtual assistant accessible through a floating character icon that opens an interactive chat interface when clicked. Skippy continuously observes and analyzes everything the user does digitally on their screen including application usage, file modifications, workspace interactions, and digital behavior patterns, transforming raw activity into meaningful context while keeping all analysis completely local on the user's device with zero network access except through controlled encrypted channels. Skippy helps users understand their own work patterns, guides them toward productivity, and serves as the intelligent eyes that detect when meaningful work has been completed. Flippo is the productivity intelligence and timeline visualization agent who transforms detected work into stunning visual representations combining nodes, threads, sticky notes, texts, images, and screenshots arranged along a temporal axis showing not just time spent but actual work accomplished. Flippo distinguishes between three hours spent on GitHub versus the actual frontend component coded during those three hours, using sophisticated matrices analyzing flow state quality, distraction frequency, tool-switching patterns, and productive versus consumptive activities to generate accurate productivity scores. Flippo runs as a cloud service because visual timeline rendering and cross-device synchronization require persistent state, but receives only work summaries and productivity metrics, never raw screen data or keystrokes. Snooks is the viral content strategist and social media growth expert who generates top-level scripts, SEO-optimized articles, hook-driven threads, and viral-ready content either from user's completed work or from scratch based on topics the user specifies. Snooks possesses deep expertise in platform algorithms, viral mechanics, engagement psychology, and concrete growth hacking strategies that actually work rather than generic advice, implementing real tactics that can sky-rocket reach when executed correctly. Snooks runs as an edge function for sub-50-millisecond response times globally and for enhanced security isolation from both Skippy and Flippo. This separation means compromising Flippo or Snooks provides no access to user screen data or workspace content, while compromising Skippy on a single user's device affects only that user and cannot pivot to other users. The agents communicate through cryptographically authenticated channels where every message includes a signature that the receiving agent verifies before processing, preventing adversaries from injecting forged messages even if they control network infrastructure.

The threat model includes supply chain attacks because MAXIM's desktop application includes numerous dependencies from NPM for the frontend, PyPI for the backend, and crates.io for the Rust components. Each dependency is a potential attack vector where a malicious maintainer could inject code that exfiltrates data or creates backdoors. The defense includes dependency pinning where exact versions are specified in lock files ensuring builds are reproducible and preventing automatic updates that could introduce malicious code. All dependencies undergo automated security scanning using tools like Snyk that check for known vulnerabilities and alert when updates are available. Critical dependencies with access to sensitive operations undergo manual code review before being incorporated. The build process uses reproducible builds where building the same source code twice produces bit-for-bit identical binaries allowing anyone to verify that distributed binaries match the claimed source code. The desktop application is code-signed using hardware-backed certificates where the signing key exists only in a hardware security module and cannot be extracted even if the build server is compromised.

The architecture implements privilege separation where processes run with minimum necessary privileges enforced by operating system security mechanisms. On Linux this uses SELinux or AppArmor mandatory access control policies that specify exactly which system calls each process can invoke, which files it can access, and which network connections it can establish. On macOS this uses sandbox entitlements declared in the application's entitlements file that limit file system access to specific directories and network access to specific domains. On Windows this uses AppContainer isolation that places processes in separate containers with restricted capabilities. These OS-level restrictions mean that even if an attacker achieves code execution within a process through a vulnerability exploit, they cannot escape the sandbox to access resources outside the declared permissions. The restrictions are enforced by the kernel which is far more thoroughly audited and hardened than application code.

## SKIPPY AGENT: THE SEEING AND UNDERSTANDING INTELLIGENCE WITH CHAT GUIDANCE

Skippy is the seeing and understanding agent who serves as the user's intelligent virtual assistant, combining screen observation capabilities, contextual analysis, workspace monitoring, and interactive chat guidance. Skippy's implementation prioritizes privacy and security over convenience in every design decision because Skippy observes the most sensitive data in the entire system including everything the user does on their screen. Skippy runs as three separate operating system processes that communicate through inter-process communication channels, implementing defense-in-depth where compromising one process does not provide access to capabilities of other processes. The three processes are Screen Observer which has display capture permissions but zero network access, Context Analyzer which has network access to communicate with Ollama but zero screen access, and Chat Interface which renders the floating character and handles user interaction but has no access to raw screen data. These processes run under separate user accounts on the operating system level so privilege escalation is required to compromise multiple processes simultaneously.

The Screen Observer process implements all digital activity monitoring functionality including screen content analysis, application usage tracking, file modification detection, workspace integration monitoring through OAuth APIs, and behavioral pattern recognition. The process runs with permissions to capture screen content through platform-specific APIs including the Screen Capture API on macOS which requires explicit user permission granted through system preferences, the Desktop Duplication API on Windows which captures the desktop in real-time, and X11 screenshot capabilities on Linux. Critically, the Screen Observer has zero network access enforced through operating system firewalls. On Linux this is implemented through iptables rules that drop all outbound packets from this process's user ID. On macOS this uses the Application Firewall configured to block all network connections from this specific executable. On Windows this uses Windows Firewall rules blocking the process. The network isolation means that even if an attacker injects malicious code into this process through a vulnerability, the code cannot exfiltrate screen captures or activity data over the network because the operating system kernel blocks all network operations.

The screen capture happens at intelligent intervals rather than continuously to balance awareness with performance and privacy. Skippy captures screenshots every 30 seconds when the user is actively working, detected through keyboard and mouse activity monitoring. When no activity is detected for 5 minutes, capture pauses entirely until activity resumes. Each screenshot undergoes on-device analysis using computer vision models running in Ollama including OCR for text extraction using Tesseract OCR engine, UI element detection using YOLO object detection identifying buttons, input fields, menus, and other interface components, application identification through window title and process name extraction, and content classification determining whether the screen shows productive work, communication, browsing, or entertainment. The analysis results are structured into activity logs containing timestamp, active application, detected UI state, extracted text snippets truncated to preserve privacy, and activity classification. The raw screenshots are discarded immediately after analysis and never stored to disk preventing forensic recovery.

The Screen Observer communicates with other Skippy processes exclusively through memory-mapped IPC channels that exist entirely in RAM and never touch the network stack. The IPC implementation uses POSIX shared memory on Unix-like systems or named shared memory on Windows where a memory region is created that multiple processes can access. Messages written to shared memory use a structured format where each message includes a header with message type, length, timestamp, and sequence number followed by the message payload. Before writing a message, the sender computes an Ed25519 digital signature over the entire message including header and payload using the sender's private signing key. The signature is appended to the message. When the receiver reads a message from shared memory, it first verifies the signature using the sender's public key which is embedded in the receiver's binary at compile time. If signature verification fails, the message is discarded and an alert is logged indicating potential tampering. This cryptographic authentication prevents an attacker who achieves local code execution from forging messages to other processes even though the shared memory is accessible.

The signing keys are generated during first-time setup through a key generation ceremony where each process generates its own Ed25519 key pair using a cryptographically secure random number generator. The private keys are stored in the operating system's secure storage with access restricted to the owning process. On macOS this uses the Keychain with kSecAttrAccessibleWhenUnlockedThisDeviceOnly protection meaning the key is accessible only while the device is unlocked and cannot sync to other devices via iCloud. On Windows this uses DPAPI Data Protection API which encrypts the key with keys derived from the user's login credentials and device-specific secrets. On Linux this uses the kernel keyring with session keyring scope meaning the key exists only while the user session is active. The public keys are exchanged during setup and embedded into each process's binary so runtime configuration cannot override them, preventing attacks where adversaries modify configuration files to substitute their own keys.

The Screen Observer also monitors workspace integrations through OAuth-connected services like Notion, Figma, and GitHub. The OAuth flow begins when the user clicks "Connect Integration" in the desktop application UI. The Screen Observer generates a cryptographically random state parameter using the operating system's secure random number generator, stores this state in memory associated with the pending authorization, and constructs an authorization URL pointing to the integration provider's OAuth endpoint with query parameters including client_id identifying MAXIM as the requesting application, redirect_uri pointing back to the local desktop application using a localhost URL with dynamically allocated port, scope specifying the permissions being requested like read-only access to files or read access to repositories, and state containing the random value generated earlier. The URL opens in the user's default web browser which navigates to the integration provider's authorization page.

The user reviews the requested permissions and clicks authorize on the provider's website. The provider redirects to the redirect_uri with query parameters including code containing a single-use authorization code and state containing the same state value sent in the original request. The Screen Observer listens on the localhost port using a temporary HTTP server implemented with Tokio in Rust. When the redirect arrives, the server extracts the code and state parameters, verifies that state matches the value stored earlier preventing cross-site request forgery attacks where adversaries trick users into authorizing integrations for attacker-controlled applications, and if verification succeeds, exchanges the authorization code for an access token by making a POST request to the provider's token endpoint with parameters including code, client_id, client_secret which is stored encrypted in the application binary, redirect_uri which must match the original redirect_uri, and grant_type with value authorization_code. The provider validates the code and client credentials and returns a response containing access_token, optional refresh_token, and expires_in indicating token lifetime.

The access token is immediately written to hardware-backed secure storage rather than being kept in memory or stored in files. The implementation uses platform-specific secure storage APIs. On macOS the code calls SecItemAdd from the Security framework with parameters including kSecClass set to kSecClassGenericPassword, kSecAttrAccount set to a unique identifier for this integration like "maxim.notion.token", kSecAttrService set to "MAXIM", kSecValueData containing the token as bytes, and kSecAttrAccessible set to kSecAttrAccessibleWhenUnlockedThisDeviceOnly. This stores the token in the Keychain protected by the Secure Enclave hardware security module where the decryption key exists only in hardware and cannot be extracted through software. On Windows the code calls CryptProtectData from the DPAPI which encrypts the token with keys derived from the user's login credentials and machine-specific secrets, storing the encrypted blob in the Windows Credential Manager. On Linux the code calls add_key from the kernel keyring API which stores the token in kernel memory with access restricted to the process's session keyring, or if TPM hardware is available, uses tpm2-tools to seal the token to the TPM where it can only be unsealed on that specific hardware.

When the Screen Observer needs to use the access token to call an integration API, it requests the token from secure storage. The operating system decrypts the token using hardware-backed keys and provides it to the process for exactly one API call, after which the token is immediately re-encrypted and removed from memory. The token never exists as plaintext in memory for more than a few milliseconds, dramatically reducing the window for memory scraping attacks. The API call to the integration provider includes the access token in the Authorization header using Bearer token format. The response is processed to extract the relevant data. For Notion this means page content, metadata, and modification timestamps. For Figma this means file names, thumbnails, version history, and sharing status. For GitHub this means repository names, commit messages, pull request titles, and release tags. The extracted data is structured into a standardized internal format with fields for source integration, resource type, resource id, title, content, timestamp, and significance indicator.

The data undergoes privacy filtering before being passed to other processes. The privacy filter implements a configurable blocklist where users can specify terms that should never appear in any data leaving the Screen Observer. This is critical for consultants working under NDAs who want Skippy to understand their work but cannot allow client names to appear in any logs or generated content. The blocklist is stored encrypted with the user's keys and loaded at startup. The filter performs exact string matching on the extracted data looking for any blocklist terms. It also performs fuzzy matching using Levenshtein distance to catch common misspellings or variations. If any term matches, the entire data record is flagged as confidential and processing is halted. The user receives a notification showing which term was detected and asking whether to exclude this record from analysis or remove the term and proceed. This explicit user control prevents data leaks from automated processing.

The Context Analyzer process receives activity summaries from the Screen Observer through the IPC channel and performs semantic understanding using the local LLM. The process runs Ollama which serves a quantized LLaMA-3-8B-Instruct model loaded into memory. The Context Analyzer has network access to communicate with Ollama over localhost but critically has zero screen capture permissions and zero file system access to user documents preventing it from reading sensitive data even if compromised. The Ollama server listens on 127.0.0.1 port 11434 and accepts HTTP POST requests to the /api/generate endpoint with JSON payloads containing the prompt and generation parameters. The Context Analyzer constructs prompts that ask the LLM to interpret activity patterns and detect meaningful work completion.

The semantic understanding prompt structure asks: "Given this sequence of activities: [activity log entries], identify whether significant work was completed. Look for patterns indicating project completion such as: final commits to main branch, document marked as done, design file marked ready for review, substantial additions to documentation, completion of test suites, or shipping of features. If work was completed, summarize what was accomplished in one to two sentences without including client names or confidential project details. If no significant work was completed, respond with NO_COMPLETION." The LLM processes the activity context and either returns a work completion summary or indicates no completion. This intelligent analysis prevents false positives where Skippy would generate notifications for trivial activities while ensuring real accomplishments are recognized.

The Chat Interface process provides the interactive guidance component of Skippy through a floating character icon and chat dialog. The character icon appears as a small circular button positioned in the bottom-right corner of the screen using platform-specific overlay rendering. On macOS this uses NSPanel with floating window level ensuring the icon stays above other windows. On Windows this uses a topmost window with transparent background except for the character icon. On Linux this uses X11 override redirect windows. The icon displays Skippy's character design as an animated sprite showing idle breathing animation when inactive and attention-getting bounce animation when new insights are available. Clicking the icon opens the chat dialog sliding in from the right side of the screen with spring animation.

The chat dialog implements a conversational interface where users can ask Skippy questions about their work patterns, request productivity insights, ask for task recommendations, or seek clarification about detected activities. Example interactions include: User asks "What did I accomplish today?" and Skippy responds with "You completed the frontend authentication component, adding login and signup flows with input validation. You also reviewed and merged two pull requests. Total focused work time was 4 hours and 23 minutes with a flow score of 87 out of 100." User asks "Why is my flow score only 87?" and Skippy explains "Your flow was interrupted 7 times by Slack notifications and you switched between projects 3 times within an hour, fragmenting your attention. Try blocking notification time or using focus mode for longer uninterrupted work periods." User asks "What should I work on next?" and Skippy suggests "Your timeline shows the backend API integration is pending for 2 days. You have 3 hours before your next meeting, which is sufficient time to make progress on the authentication endpoints." This conversational guidance helps users understand their productivity patterns and make informed decisions about time allocation.

The chat conversations use the LLM for natural language understanding and generation but include guard rails preventing Skippy from accessing real-time screen data during conversations. When a user asks about their activities, Skippy queries the activity database for historical summaries rather than requesting live screen captures. This separation ensures conversations cannot be exploited to exfiltrate sensitive information even if an adversary tricks the user into asking revealing questions. The chat history is stored locally in the encrypted SQLite database with retention limited to 30 days after which old conversations are automatically purged to minimize data footprint.

## FLIPPO AGENT: PRODUCTIVITY INTELLIGENCE WITH VISUAL TIMELINE CREATION

Flippo is the productivity intelligence and timeline visualization agent responsible for transforming detected work into stunning visual representations, calculating accurate productivity metrics, generating productivity scores, and creating the MAXIM Wrapped reports. Flippo's implementation runs entirely as cloud services because visual timeline rendering requires significant computational resources for image generation, cross-device synchronization requires persistent state, and the visual assets must be accessible from web browsers and mobile devices. However, Flippo's architecture is designed so that even with complete backend compromise, user screen data and raw activity logs remain inaccessible. Flippo receives only work completion summaries, productivity metrics, and timestamps, never screenshots, raw text, or detailed application usage logs.

Flippo's backend is implemented as a FastAPI application written in Python 3.11 running on Railway's global infrastructure which provides automatic HTTPS, environment variable management, health checks, and zero-downtime deployments. The application is structured in layers where routers handle HTTP requests and parse parameters, services contain business logic and orchestrate operations, repositories manage database access through Prisma ORM, and workers execute background tasks through Celery. This layered architecture separates concerns making the codebase maintainable and testable. Routers are stateless and can scale horizontally by running multiple instances behind a load balancer. Services are pure functions that accept inputs and produce outputs without side effects making them easy to test. Repositories encapsulate all SQL knowledge preventing SQL injection vulnerabilities through improper query construction.

The authentication system for Flippo's API uses JWT tokens signed with RS256 asymmetric cryptography. The signing process uses a private key stored in a Hardware Security Module on the backend infrastructure where the key cannot be extracted even with root access to the server. The public key is distributed to clients and used to verify token signatures. When a user authenticates through the desktop client, the Cloud Communicator requests a token from the /auth/login endpoint by sending the user's ID and a proof of possession for their passkey credential. The backend verifies the proof, generates a JWT with claims including sub containing the user ID, iat containing issued-at timestamp, exp containing expiration timestamp set to 30 minutes from issuance, and jti containing a unique token ID. The token is signed with the HSM private key and returned to the client.

The timeline visualization generation is Flippo's signature capability that transforms abstract work logs into tangible visual narratives. When Skippy detects a work completion event, it sends a timeline_update event to Flippo containing the work summary, timestamp, duration, application used, and integration source. Flippo processes this event by creating a new timeline node and generating visual assets. Timeline nodes are represented as a directed acyclic graph where each node represents a discrete work accomplishment and edges represent temporal or causal relationships between accomplishments. Nodes contain structured data including unique node_id, timestamp indicating when the work was completed, title summarizing what was accomplished, detailed_description providing more context, work_type categorizing the accomplishment such as coding, design, writing, or review, duration_minutes indicating how long the work took, flow_score from 0 to 100 indicating quality of focused attention during that work period, and attached_evidence including screenshots, code diffs, or document excerpts that prove the work was done.

The visual representation uses a hybrid approach combining SVG for the timeline structure and Canvas2D for rendering complex elements. The timeline axis is drawn as an SVG path with dates marked along the bottom and work nodes positioned above the timeline at their corresponding timestamps. Each node is rendered as a circular element with radius proportional to the significance of that accomplishment. Small routine tasks have radius 20 pixels while major project completions have radius 60 pixels. The nodes are colored using a gradient that maps from cool blues for analytical work like coding to warm ambers for creative work like design to greens for collaborative work like reviews and meetings. This color coding allows users to visually identify the type of work at a glance.

Inside each node is a thumbnail representation of the evidence. For code work, this shows a miniature syntax-highlighted code snippet. For design work, this shows a thumbnail of the designed asset. For writing work, this shows the first few words in the document's font. These thumbnails are generated server-side using Puppeteer for web content, Sharp for image manipulation, and Pygments for code syntax highlighting. The thumbnail generation happens asynchronously as a background job to avoid blocking timeline rendering. When a user first views their timeline, nodes appear with placeholder thumbnails that are replaced by actual thumbnails as they generate creating a progressive loading experience.

Nodes are connected by curves called threads that show relationships between related work. When consecutive work items are part of the same project, they are connected by a thread. The thread is rendered as a Bezier curve with control points calculated to create smooth flowing connections rather than straight lines. The threads are colored to match the average color of the nodes they connect creating visual continuity. Threads are annotated with sticky notes showing contextual information like "2-hour gap: lunch break" or "switched to different project" helping users understand discontinuities in their work flow. These annotations are positioned along the thread using SVG text elements with backgrounds creating readable labels.

The timeline includes text annotations as well where users can add notes explaining context, recording decisions made, or capturing insights. Notes are created by clicking the "Add Note" button which opens a text editor overlaying the timeline. Users type their note, choose a position along the timeline, and save. The note appears as a sticky-note styled div with yellow background, handwritten-style font, slight rotation for organic appearance, and a pin icon indicating it is pinned to that specific moment in time. Notes support Markdown formatting allowing users to include bold text, links, or bullet lists providing flexibility for different note-taking styles.

Screenshots serve as proof of work in the timeline. When Skippy detects a significant accomplishment, it captures a screenshot at that moment before discarding raw screen data. This screenshot is uploaded to Flippo where it is stored in object storage like S3 with access restricted to the owning user. The screenshot appears in the timeline at the corresponding timestamp as a rectangular card with a preview thumbnail. Clicking the thumbnail opens a lightbox showing the full-resolution screenshot. The screenshots are automatically analyzed using computer vision to extract key elements like code editors, design tools, or browser tabs making them searchable. Users can search their timeline with queries like "show me when I was working on the authentication component" and Flippo returns all timeline nodes and screenshots containing relevant visual or textual evidence.

The productivity scoring algorithm implemented by Flippo is sophisticated and multi-dimensional going far beyond simple time tracking. The algorithm considers multiple factors including focused work time calculated by measuring continuous periods of activity in productivity applications without switching to distracting applications, flow state quality measured by analyzing the velocity of productive actions where rapid consistent activity indicates deep focus, distraction frequency calculated by counting context switches between applications particularly switches from productive tools to communication or entertainment applications, productive versus consumptive activity ratio distinguishing between creating content versus consuming content like coding versus reading documentation, and tool mastery indicated by efficient keyboard shortcut usage and minimal time in menus or searching for features. Each factor contributes to an overall productivity score from 0 to 100 where scores above 85 indicate exceptional focus and productivity, scores 70 to 85 indicate good productivity with minor distractions, scores 50 to 70 indicate moderate productivity with significant distractions, and scores below 50 indicate poor productivity with work constantly interrupted.

The score calculation uses weighted averages where focused work time has weight 0.3, flow state quality has weight 0.3, distraction frequency has inverse weight 0.2, productive ratio has weight 0.15, and tool mastery has weight 0.05. These weights are tuned based on research into productivity and cognitive performance and can be customized by users who want to emphasize different factors. For example, a creative professional might increase the flow state weight because creative work requires deep immersion, while a customer support representative might decrease distraction frequency weight because their job naturally involves many application switches to help different customers. The algorithm also learns personalized patterns over time identifying what productivity looks like specifically for each user rather than applying generic standards.

Flippo distinguishes between time spent and actual work done which is critical for accurate productivity assessment. Consider a user who has GitHub open for 3 hours. The naive approach counts this as 3 hours of coding work. Flippo's approach analyzes what actually happened during those 3 hours by examining git commits, code additions and deletions, pull request activity, and CI/CD pipeline runs. If the user made 5 commits adding 200 lines of frontend code implementing an authentication component, Flippo records "Coded authentication component with login/signup flows: 200 LOC added" and calculates that actual productive coding was approximately 1.5 hours based on typical coding velocity, with the remaining 1.5 hours spent on reading documentation, debugging, or breaks. This distinction helps users understand where their time actually goes rather than feeling puzzled why 3 hours of work yielded limited output.

The matrices used for analysis include the Eisenhower Matrix classifying work as urgent-important, urgent-not-important, not-urgent-important, or not-urgent-not-important helping users visualize whether they are spending time on high-impact work or getting trapped in urgency addiction. The Energy-Time matrix plotting work against both time of day and estimated energy expenditure helping users identify their peak performance windows and optimize scheduling. The Flow Cycle matrix tracking the progression through the four phases of the flow cycle including struggle where new information is absorbed, release where subconscious processing occurs, flow where peak performance happens, and recovery where energy is restored, helping users balance intense work with necessary recovery. These matrices are visualized as interactive charts in the dashboard allowing users to explore their productivity patterns from multiple perspectives.

Flippo's MAXIM Wrapped feature generates monthly performance reports designed to make users proud of their accomplishments and encourage social sharing. On the first day of each month, Flippo's wrapped worker queries all work completion events for each active user in the previous month. The worker calculates aggregate metrics including total work items completed, total productive time excluding breaks and distractions, average productivity score for the month, top productive day with the highest score, longest continuous flow state session, most productive time of day based on when highest-scoring work occurred, primary work type showing whether the user spent more time coding, designing, writing, or other activities, productivity trend comparing first half of month to second half, and accomplishment highlights extracting the three most significant work items completed. The worker also computes work pattern evolution by comparing activity patterns from early in the month to late in the month identifying whether the user is becoming more focused, more distracted, or maintaining consistency.

These metrics are packaged into a data structure and passed to a rendering service built with React and Three.js hosted on Vercel as a serverless function. The function generates a static HTML page with embedded JavaScript for animations. The page displays metrics with smooth counting animations where numbers increment from zero to final values using easing functions over approximately 2 seconds creating satisfying visual feedback. Three.js renders a particle background with thousands of amber dots floating upward creating a celebratory atmosphere. The timeline is rendered as an interactive 3D visualization where users can rotate and zoom exploring their month's work from different angles. The user's productivity evolution is visualized as a line graph showing the daily productivity score with annotations marking significant accomplishments and showing correlations between high-scoring days and specific work patterns.

The wrapped page is deployed to a CDN with a unique URL structure maxim.app/wrapped/[hash]/[year]/[month] where hash is a cryptographic hash of user_id preventing enumeration attacks where adversaries iterate user IDs to find wrapped pages. The page includes Open Graph metadata for rich social media previews showing the user's top metrics, making shared links visually compelling with preview images, engaging descriptions, and proper metadata. The page footer includes subtle MAXIM branding "Created with MAXIM" linking to the landing page creating viral growth loops where users sharing their wrapped reports drive awareness and traffic to MAXIM without the sharing feeling like advertising.

## SNOOKS AGENT: VIRAL CONTENT STRATEGIST WITH GROWTH HACKING EXPERTISE

Snooks is the viral content strategist and social media growth expert who generates high-level scripts, SEO-optimized articles, hook-driven content, and viral-ready materials that maximize reach and engagement. Snooks possesses deep expertise in platform algorithms including how LinkedIn's feed prioritization works, X's recommendation engine mechanics, Instagram's Explore algorithm, YouTube's recommendation system, and TikTok's For You page algorithm. Snooks understands concrete growth hacking strategies including the optimal posting times per platform, the hashtag strategies that actually drive discovery, the content structures that maximize dwell time signaling value to algorithms, the call-to-action placements that drive clicks without appearing spammy, and the viral loop mechanics that encourage sharing. Snooks generates content either based on work the user has completed as detected by Skippy or from scratch based on topics the user specifies making Snooks flexible for both showcasing accomplishments and creating thought leadership content.

Snooks' implementation runs as a serverless edge function on Cloudflare Workers for sub-50-millisecond global latency ensuring content generation feels instant regardless of user location. Snooks' architecture provides security isolation from both Skippy and Flippo meaning compromising Snooks does not provide access to screen data, activity logs, or even timeline visualizations. Snooks receives only high-level work summaries that users explicitly approve for content generation plus any additional context the user provides through the content creation interface. The edge function is written in TypeScript and deployed to Cloudflare's global network using Wrangler. The function receives content generation requests as HTTP POST requests to /content/generate with JSON body containing optional work_summary if generating from completed work, topic if generating from scratch, platform specifying the target social media platform, content_type specifying the desired format like thread, article, carousel script, or video script, desired_length, and optional style_preferences.

The content generation process begins with Snooks analyzing the input to understand the core message, identify key angles that would resonate with target audiences, determine the optimal content structure for maximum engagement on the specified platform, and select growth hacking elements to incorporate. For platform-specific optimization, Snooks implements different strategies per platform. For LinkedIn, Snooks generates long-form posts of 1300 to 1900 characters which hit the sweet spot of LinkedIn's algorithm showing strong preference for longer content that keeps users on the platform, structures content with a hook in the first 2 lines before the See More break capturing attention in the feed, includes 3 to 5 relevant hashtags but avoids over-hashtagging which appears spammy, incorporates storytelling elements because personal narratives drive emotional engagement leading to higher comment rates, asks thought-provoking questions in the closing paragraph encouraging comments which signal high-quality content to the algorithm, and tags 1 to 2 relevant connections when appropriate to expand reach through their networks.

For X threads, Snooks generates 6 to 12 tweet sequences where the first tweet functions as a standalone hook that delivers value even if readers don't continue ensuring strong engagement metrics on the first tweet which determines whether X promotes the thread, each subsequent tweet advances the narrative with one idea per tweet maintaining momentum, threads use line breaks and emojis strategically for visual scanning because X's fast-scrolling culture rewards scannable content, the final tweet includes a call-to-action like "Follow me for more insights on [topic]" or "Share this if you found it valuable" encouraging profile visits and retweets, threads incorporate 1 to 2 relevant hashtags only in later tweets not the first tweet because hashtags in the first tweet can reduce engagement, and threads are published at optimal times which for most audiences is Tuesday through Thursday between 10 AM and 3 PM when engagement is highest.

## THE INTER-AGENT ORCHESTRATION AND WORKFLOW COORDINATION

The three agents operate independently but coordinate through well-defined workflows that implement the complete journey from detecting work through generating content to publishing and analyzing performance. The workflows are event-driven where each agent reacts to events produced by other agents, with events communicated through message passing rather than shared state. This loose coupling means agents can be updated independently without breaking the overall system as long as message interfaces remain compatible.

The primary workflow begins when Skippy's Screen Observer detects that meaningful work has been completed. The detection happens continuously as Skippy monitors screen content and workspace integrations. When Skippy's Context Analyzer determines that an accomplishment occurred, it creates a work_completed event containing integration_type, resource_title if from an integration, work_summary generated by the local LLM describing what was accomplished without sensitive details, detected_at timestamp, duration_estimate indicating approximately how long the work took, and work_type categorizing the work such as coding, design, writing, or review. The event is written to a local event queue implemented as a SQLite table with columns for event_id, event_type, payload as JSON, created_at, and processed_at initially null. Writing to the event queue is transactional ensuring events are not lost if Skippy crashes.

The event is forwarded to both Flippo and Snooks through the Cloud Communicator process. For Flippo, the event triggers timeline node creation where Flippo generates visual assets and calculates productivity metrics. For Snooks, the event is added to the content opportunities queue where it waits for user review. The user reviews opportunities in the Command Center where each opportunity appears as a card showing what work was completed and suggesting "Generate content about this". Clicking the card opens the content generation interface where the user selects target platforms and content types then clicks Generate.

The content generation request is sent to Snooks' edge function which processes the request and returns generated content variations. The content appears in the content editor where the user can review, refine through conversational editing with Snooks, and approve for publishing. During the refinement conversation, the user might say "make this more technical" or "remove the section about implementation details" or "add a story about why this matters" and Snooks regenerates the content incorporating the feedback. The conversation continues until the user is satisfied.

When the user clicks Approve, the content undergoes final compliance verification where Snooks runs platform policy checks, PII detection, and link safety validation as described earlier. If checks pass, the content advances to the scheduling interface where the user chooses to publish immediately or schedule for later. Flippo provides posting time recommendations based on historical engagement data showing "Tuesday 9 AM typically gets 40% higher engagement for your LinkedIn content" allowing data-driven scheduling decisions. After scheduling, the content enters the queue and publishes automatically at the scheduled time.

After publishing, Flippo's analytics worker collects engagement metrics by polling platform APIs. The metrics are stored in the Analytics table and used to calculate engagement rates, identify top-performing content, and refine posting time recommendations. Flippo's learning system analyzes which content angles performed best, which posting times drove highest engagement, and which calls-to-action generated most clicks feeding these insights back into Snooks' generation prompts and Flippo's scheduling recommendations creating a continuous improvement loop.

## THE LOCAL DATABASE ARCHITECTURE AND ENCRYPTION AT REST

The local database stores all sensitive user data including activity logs, work summaries, chat conversations with Skippy, integration credentials references, and audit logs. The database must remain secure even if an attacker achieves file system access to the user's device. The protection is implemented through encryption at rest where the entire database file is encrypted and decryption keys are derived from the user's passkey credential. Without the passkey, the database file is useless ciphertext.

The database is implemented using SQLCipher which is SQLite with transparent AES-256 encryption. The database file appears as encrypted blob to the file system. When the application opens the database, it provides the encryption key through SQLCipher's PRAGMA key command. SQLCipher decrypts pages on demand as they are read and re-encrypts pages when they are written. The encryption uses AES-256 in CBC mode with HMAC-SHA512 for authentication preventing tampering. Each page is encrypted with a unique initialization vector derived from the page number preventing block-level attacks.

The encryption key is derived from the user's passkey credential through Argon2id key derivation function. Argon2id is a memory-hard KDF resistant to GPU and ASIC cracking attacks. The parameters are memory cost 64 MB meaning the KDF requires 64 megabytes of RAM to compute, time cost 3 iterations meaning the KDF runs its mixing function 3 times, parallelism 4 threads allowing the KDF to utilize multi-core processors, and output length 32 bytes producing a 256-bit key suitable for AES-256. These parameters make brute-force attacks computationally infeasible while remaining fast enough that legitimate users experience acceptable latency.

The passkey credential provides the input to Argon2id. During first-time setup, the user creates a passkey through the WebAuthn ceremony. The browser generates a cryptographic key pair where the private key is stored in the device's secure hardware like TPM or Secure Enclave and the public key is sent to MAXIM's backend. The credential includes a credential_id uniquely identifying this credential. The credential_id combined with a user-specific salt is used as input to Argon2id. The salt is generated randomly during setup and stored in plaintext alongside the encrypted database. The salt prevents rainbow table attacks where adversaries precompute hashes for common passphrases.

Each time the application starts, the user authenticates with their passkey through WebAuthn. The authentication ceremony proves possession of the private key without revealing it. After successful authentication, the application reconstructs the Argon2id input from the credential_id and stored salt, runs the KDF to derive the database encryption key, and opens the database with this key. If the key is incorrect, SQLCipher fails to decrypt the database and returns an error. The application detects this and prompts for re-authentication.

The database schema is designed for security and integrity. The Activity_Log table stores screen activity summaries with timestamp, application, activity_type, and truncated_context ensuring raw screen data is never persisted. The Work_Events table stores detected work completions with work_summary, work_type, duration, and flow_score. The Integration_Tokens table stores references to tokens in the hardware keystore rather than the tokens themselves. The Chat_History table stores conversations with Skippy with retention limited to 30 days. The Audit_Log table implements append-only logging with entries for every security-relevant action including authentication, data access, configuration changes, and compliance overrides.

The append-only property is enforced through database triggers that prevent updates and deletes on the Audit_Log table. Attempting to modify or delete a log entry raises an error. The table has an autoincrement primary key ensuring each entry has a unique sequential ID. Each entry includes a hash of the previous entry computed as SHA-256 of the previous entry's entire row serialized as JSON. This creates a Merkle chain where each entry cryptographically links to all previous entries. Tampering with any entry breaks the chain making detection trivial. Users can export the audit log and verify the hash chain independently.
